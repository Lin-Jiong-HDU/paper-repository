# AppWorld: A Controllable World of Apps and People for Benchmarking Interactive Coding Agents

## 基本信息

- **标题**: AppWorld: A Controllable World of Apps and People for Benchmarking Interactive Coding Agents
- **作者**: Harsh Trivedi, Tushar Khot, Mareike Hartmann, Ruskin Manku, Vinty Dong, Edward Li, Shashank Gupta, Ashish Sabharwal, Niranjan Balasubramanian
- **发表日期**: 2024-07-26
- **arXiv ID**: 2407.18901
- **arXiv 链接**: <https://arxiv.org/abs/2407.18901>
- **领域/关键词**: 自主智能体、代码生成、基准测试、API 调用、工具使用、交互式编码

## 一句话总结
>
> AppWorld 构建了一个由 9 个日常应用和 457 个 API 组成的可控执行环境，以及包含 750 个任务的基准测试套件，专门用于评估需要通过编写复杂代码与多个应用交互的自主编码智能体。

## 论文精读

### 背景与前置知识

理解本论文需要掌握以下基础概念：

- **自主智能体（Autonomous Agent）**：能够根据自然语言指令，自动规划并执行一系列操作来完成任务的 AI 系统。与传统的单次问答不同，智能体需要在环境中反复交互——观察结果、调整策略、继续执行。

- **工具使用/函数调用（Tool Use / Function Calling）**：大语言模型通过调用外部工具（如 API）来完成超出其自身能力的任务。例如查询数据库、发送消息、下单购物等。现有的基准测试大多只涉及简单的线性 API 调用序列。

- **交互式代码生成（Interactive Code Generation）**：智能体不仅要调用 API，还需要编写包含复杂控制流（循环、条件判断、异常处理）的代码，并且在与环境交互的过程中迭代地修改和完善代码。

- **沙箱执行环境（Sandboxed Execution Environment）**：一个安全隔离的代码运行环境，智能体在其中执行代码不会影响真实世界。AppWorld 就是这样一种环境。

- **附带损害（Collateral Damage）**：智能体在完成主要任务时，意外修改或破坏了与任务无关的数据。例如，任务是给 Alice 转账，但智能体不小心删除了 Bob 的联系人信息。

### 核心思想详解

可以用一个生活中的类比来理解 AppWorld：想象你有一个数字助手，你需要让它帮你完成"查看冰箱里的食材清单（在笔记应用中），然后根据缺少的物品在购物应用下单，最后通过消息应用告诉家人买了什么"这样的跨应用任务。

现有的基准测试存在一个根本性缺陷：它们假设智能体只需要按顺序调用几个 API 就能完成任务。但现实世界中的日常数字任务远比这复杂——它们需要跨多个应用操作，需要编写包含循环、条件判断的代码，需要在执行过程中根据反馈动态调整。

AppWorld 的核心思想是构建一个**完整的、可执行的、可控的数字世界**：

1. **AppWorld Engine（引擎）**：这是一个包含 9 个日常应用（笔记、消息、通讯录、购物、银行、音乐、电影、天气、支出追踪）的执行环境，共有 457 个 API，模拟了约 100 个虚构用户的真实数字生活。引擎约 6 万行代码，每个应用都有完整的后端逻辑和数据库。

2. **AppWorld Benchmark（基准）**：包含 750 个自然、多样且具有挑战性的任务。每个任务不仅要求智能体调用 API，还要求编写具有复杂控制流的代码。基准约 4 万行代码，包括任务描述、参考解答和自动化测试。

3. **评估机制**：使用基于状态的单元测试（state-based unit tests）进行评估。这种评估方式允许不同的完成任务路径（殊途同归），同时检测附带损害（确保智能体没有意外修改无关数据）。

### 方法逐步拆解

**第一步：构建 AppWorld Engine**

- 设计 9 个日常应用，每个应用模拟真实世界的功能：Contacts（通讯录）、Messaging（消息）、Notes（笔记）、Shopping（购物）、Banking（银行）、Music（音乐）、Movies（电影）、Weather（天气）、Expense Tracking（支出追踪）
- 为每个应用实现完整的后端 API，共计 457 个可调用的 API
- 使用 SQLite 数据库存储应用状态，每个应用有独立的数据表
- 填充约 100 个虚构用户的数据，包括他们的联系人、消息历史、购物记录、银行账户、音乐播放列表等，确保数据之间的关联关系真实合理

**第二步：创建 AppWorld Benchmark 任务**

- 设计 750 个任务，分为 "normal"（一般难度）和 "challenge"（高难度）两个级别
- 每个任务包含：自然语言指令、ground truth 代码解答、测试用例
- 任务需要跨多个应用操作，例如"查看 Alice 的生日（通讯录），给她买一份她最喜欢的歌手的专辑（音乐+购物），然后发消息告诉她（消息）"
- 任务的参考解答代码平均包含循环、条件判断、异常处理等复杂控制流

**第三步：设计评估框架**

- 采用基于状态的单元测试：执行智能体生成的代码后，检查应用的最终状态是否满足预期
- 支持多解性：不同的代码实现方式只要最终状态正确都算通过
- 附带损害检测：除了验证任务目标是否达成，还检查无关数据是否被意外修改
- 将评估分为两个维度：completion（任务完成度）和 collateral damage（附带损害率）

**第四步：实验评估**

- 测试多个 LLM：GPT-4o、GPT-4-Turbo、Claude-3、Gemini-1.5-Pro、Llama-3-70B 等
- 使用 ReAct 风格的智能体框架，让模型在多轮交互中生成代码、观察执行结果、调整代码
- 分析错误类型：API 调用错误、逻辑错误、控制流错误、附带损害等

### 关键公式/算法解读

本论文的核心不在于数学公式，而在于系统设计和评估框架。以下是几个关键的技术设计：

- **任务评估公式**：对于每个任务 t，智能体的输出代码 c 在 AppWorld Engine 中执行后，得到最终应用状态 S_final。评估通过测试函数 T(S_final) 来验证：
  - Completion Check：T_primary(S_final) 检查主要目标是否达成
  - Collateral Check：T_collateral(S_initial, S_final) 检查无关数据是否被修改
  - 只有两者都通过，任务才算完全成功

- **智能体交互流程**：
  1. 接收自然语言任务描述
  2. 生成 Python 代码调用 AppWorld API
  3. 在沙箱环境中执行代码
  4. 观察执行结果（输出、错误信息）
  5. 根据反馈决定是否需要修改代码并重新执行
  6. 重复直到任务完成或达到最大交互轮数

- **API 设计模式**：每个 API 遵循统一的设计模式——接收结构化参数、返回 JSON 格式结果、维护应用状态的一致性。例如 `search_contacts(name="Alice")` 返回匹配的联系人列表，`send_message(to=contact_id, body="Hello")` 发送消息并更新消息历史。

### 实验设计分析

**为什么选择这些应用和数据集？**

- 9 个应用覆盖了日常生活中最常见的数字操作场景：沟通（消息、通讯录）、信息管理（笔记、支出追踪）、娱乐（音乐、电影）、日常生活（购物、天气、银行）
- 这些应用的组合能够模拟真实的跨应用任务链，比单应用场景更具挑战性
- 约 100 个虚构用户的数据确保了任务的多样性和真实性

**主要指标：**

- **Completion Rate（完成率）**：任务目标达成的比例
- **Collateral Damage Rate（附带损害率）**：任务中产生不期望的副作用的 proportion
- **Strict Success Rate（严格成功率）**：同时满足完成率和无附带损害的比例

**关键实验结果：**

- **GPT-4o** 在 normal 任务上完成率约 49%，challenge 任务约 30%
- 其他模型（GPT-4-Turbo、Claude-3、Gemini-1.5-Pro）比 GPT-4o 至少低 16 个百分点
- 附带损害是一个普遍问题：即使任务完成了，很多智能体也会意外修改无关数据
- 主要错误类型包括：API 使用错误（参数错误、调用顺序错误）、逻辑推理错误（对任务需求理解偏差）、控制流错误（循环/条件判断逻辑不对）

**为什么这些结果有意义？**

这些结果表明，即使是最先进的 LLM，在面对需要编写复杂代码并与多个应用交互的任务时，表现仍然远远不够理想。这揭示了现有智能体在真实世界应用场景中的局限性，也为未来的研究指明了方向。

## 问题与动机

现有的工具使用基准测试（如 ToolBench、API-Bank、Gorilla 等）存在以下关键局限：

1. **过度简化**：大多数任务只需要按顺序调用 2-3 个 API，不需要复杂的控制流
2. **缺乏交互性**：任务是一次性完成的，不需要根据执行反馈迭代调整
3. **评估不充分**：通常只检查最终输出是否匹配预期，不检查过程中是否产生了附带损害
4. **数据不真实**：API 和数据往往是合成的，缺乏真实世界的复杂性和关联性

AppWorld 的动机是创建一个更贴近真实世界需求的基准测试，推动交互式编码智能体的发展。

## 核心方法

AppWorld 由两个核心组件构成：

**AppWorld Engine** 是执行环境，包含：
- 9 个功能完整的应用（每个都有后端逻辑、数据库、API）
- 457 个精心设计的 API，涵盖 CRUD 操作、搜索、过滤、排序、聚合等
- 约 100 个虚构用户的真实数字生活数据
- 沙箱化的代码执行环境，支持安全执行智能体生成的代码

**AppWorld Benchmark** 是测试集，包含：
- 750 个手工编写的任务，覆盖多种任务模式
- 每个任务有自然语言描述、参考代码解答、自动化测试用例
- 任务按难度分为 normal 和 challenge 两个级别
- 支持程序化评估，包括任务完成度检查和附带损害检测

## 实验结果

- **数据集**: AppWorld Benchmark（750 个任务，9 个应用，457 个 API，约 100 个用户）
- **主要指标**: Completion Rate（完成率）、Collateral Damage Rate（附带损害率）、Strict Success Rate（严格成功率）
- **关键结果**: GPT-4o 在 normal 任务上约 49% 完成率，challenge 任务约 30%；其他模型至少低 16%；附带损害是普遍问题
- **对比方法**: GPT-4o、GPT-4-Turbo、Claude-3-Opus、Claude-3-Sonnet、Gemini-1.5-Pro、Gemini-1.5-Flash、Llama-3-70B、Llama-3-8B

## 局限性

1. **应用范围有限**：虽然 9 个应用覆盖了日常场景，但真实世界的应用生态远比这复杂（社交媒体、地图导航、文件管理等均未包含）

2. **无 GUI 交互**：AppWorld 只支持 API 交互，而真实的智能体可能需要通过图形界面操作应用

3. **语言限制**：任务和 API 都是英文的，没有覆盖多语言场景

4. **静态数据**：虚构用户的数据是固定的，不会像真实世界那样持续更新和变化

5. **缺乏多用户交互**：模拟的是单用户视角，没有考虑多用户并发操作的场景

6. **没有涉及长期记忆**：任务都是独立的，没有测试智能体在多天/多任务间的记忆和持续学习能力

## 个人思考

AppWorld 是一个非常精心设计的基准测试，它准确地抓住了当前智能体评估中的一个关键空白——现有基准过于简化，不能反映真实世界中跨应用、需要复杂代码生成任务的挑战。

几个值得深入思考的点：

- **附带损害**这个概念非常有价值。在真实部署中，智能体不仅需要完成任务，还不能搞破坏。这个评估维度应该在更多基准测试中被采纳。

- 750 个任务虽然不算多，但每个任务都有手工编写的参考解答和测试用例，质量很高。基准测试的质量应该优先于数量。

- 即使是最强的 GPT-4o 也只能解决约一半的 normal 任务，说明这个领域还有很大的提升空间。特别是复杂控制流的代码生成和多步推理能力，是未来需要重点突破的方向。

- AppWorld 的可控性（所有应用都是自建的）使得评估结果更加可靠和可复现，这是一个重要的设计选择。但它与真实 API 的差异也可能影响结论的外推性。
