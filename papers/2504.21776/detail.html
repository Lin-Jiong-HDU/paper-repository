WebThinker: Empowering Large Reasoning Models with Deep Research Capability

Xiaoxi Li, Jiajie Jin, Guanting Dong, Hongjin Qian, Yongkang Wu, Ji-Rong Wen, Yutao Zhu, Zhicheng Dou 2025-04-30
arXiv:2504.21776
大型推理模型 深度研究 网络搜索 报告生成 强化学习 DPO

一句话总结

WebThinker 通过为大型推理模型集成深度网络探索器和自主"思考-搜索-起草"策略，使模型在推理过程中能自主搜索网页、导航和撰写研究报告，显著提升了复杂推理和报告生成任务的性能。

背景与前置知识

▼

理解本论文需要以下基础概念：

大型推理模型（LRM）：如 OpenAI o1、DeepSeek-R1 这类模型，具备长时间推理（long-horizon reasoning）能力。它们通过生成"思维链"（Chain of Thought）来逐步解决复杂问题。但这类模型有一个关键缺陷：它们只能依赖训练时学到的静态知识，无法在推理过程中获取新信息。

检索增强生成（RAG）：一种常见的做法是给模型接入搜索引擎，让它能检索外部信息。传统 RAG 通常是"先检索，再生成"的两步流程，模型无法在推理中间动态决定何时搜索。

Direct Preference Optimization（DPO）：一种基于偏好数据的训练方法。给定一组"好的"和"差的"回答，DPO 直接优化模型使其更倾向于生成好的回答，而不需要训练一个单独的奖励模型。

深度网络探索：不同于简单的"搜索-点击"流程，深度探索意味着模型可以在搜索结果中导航、点击链接、在页面间跳转，逐层深入地获取信息，类似人类研究者在网上追踪信息线索的方式。

核心思想详解

▼

WebThinker 的核心思想可以用一个类比来理解：把一个"闭卷考试"的学生变成一个"开卷但可以查资料"的研究者。

想象你在解答一道复杂的科学题。传统 LRM 就像一个学生只能凭记忆作答——如果记忆中没有相关知识，就会出错或瞎猜。而 WebThinker 就像允许这个学生在解题过程中随时去图书馆查资料，不仅能查，还能在图书馆里翻阅多本书、交叉比对，甚至边查资料边写研究报告。

具体来说，WebThinker 有两个创新点：

1. Deep Web Explorer（深度网络探索器）：当模型在推理中遇到知识盲区时，可以调用这个工具。它不仅能搜索，还能在搜索结果中导航——点击链接、浏览页面内容、继续搜索更多相关信息。这是一种"递归式"的深度探索，不是一次性搜索。

2. Autonomous Think-Search-and-Draft（自主思考-搜索-起草）：在报告生成模式下，模型可以在推理、信息搜索和报告撰写之间无缝切换。模型就像一个真正的研究者：先思考需要什么信息，去网上搜索，找到信息后写一段报告，发现不够再继续搜索补充，如此循环直到完成完整的报告。

方法逐步拆解

▼

步骤 1：整体框架设计

  • WebThinker 有两个工作模式：问题解决模式和报告生成模式
  • 问题解决模式专注于回答复杂问题（如 PhD 级别科学问题）
  • 报告生成模式专注于撰写全面的研究报告

步骤 2：Deep Web Explorer 工具构建

  • 模型在推理链中可以调用 Deep Web Explorer 工具
  • 该工具内部有两种基本操作：搜索引擎查询（生成查询词并获取网页列表）和页面导航（点击链接或按钮，浏览页面内容）
  • 探索器有自己的内部推理链，自主决定是否继续搜索或深入导航
  • 最终输出一个简洁的结果，填补主推理链中的知识空白

步骤 3：报告生成的写作工具

  • 除了探索工具，报告模式还配备了三个写作工具：起草、检查和编辑
  • 这些工具由一个辅助 LLM 执行，将复杂的任务编排与具体的文本操作分离开来
  • 所有探索过的网页被存储在文档记忆中，供写作时检索引用

步骤 4：偏好数据构建

  • 使用 WebThinker 在多个复杂数据集上采样不同的推理轨迹
  • 按三个优先级规则构建偏好对：(1) 正确性/报告质量优先；(2) 工具调用效率（更少的工具调用更优）；(3) 思维简洁性（更短的推理链更优）

步骤 5：迭代在线 DPO 训练

  • 利用构建的偏好数据集训练模型
  • 迭代过程：训练 → 用新模型采样 → 构建新偏好数据 → 更新参考策略 → 重复
  • 通过与环境交互的 on-policy 方式，持续优化工具使用策略

关键公式/算法解读

▼

公式 1（主推理过程）：描述了问题解决模式下的推理过程。推理链的每一步都依赖于之前的推理步骤和所有已完成的 Deep Web Explorer 返回的结果。最终答案的生成依赖于完整的推理链和原始问题。

公式 2（深度探索器内部过程）：描述探索器内部的推理过程。每一步的决策都基于当前已获取的网页内容（动态变化），决定是继续搜索/导航还是输出结果。

公式 3（报告生成过程）：辅助 LLM 根据编辑指令、从文档记忆中检索的相关文档和当前报告状态来更新报告内容。

公式 4（DPO 损失函数）：标准 DPO 损失：增大偏好轨迹的概率，降低非偏好轨迹的概率。β 控制偏离参考策略的程度，σ 是 sigmoid 函数。通过迭代在线方式，模型持续优化工具使用策略。

实验设计分析

▼

为什么选这些数据集？

  • GPQA：测试 PhD 级别的科学问答能力，需要专业知识，最能体现"知识密集型推理"
  • GAIA：测试通用 AI 助手能力，包含多层级复杂信息获取任务
  • WebWalkerQA：专门测试深度网页导航和信息提取能力，直接对应 WebThinker 的核心能力
  • HLE：跨学科极端难题，需要高级搜索和推理能力
  • Glaive：测试开放式推理问题的报告生成能力

实验结果说明什么？

  • 在 GPQA 上，WebThinker-32B-RL 达到 70.7% 准确率，超越 QwQ-32B 的 64.1%，接近 DeepSeek-R1-671B 的 74.2%
  • 在 GAIA 上达到 48.5% 平均分，在 32B 模型中显著领先
  • 在 WebWalkerQA 上达到 46.5%，远超所有 RAG 方法和 Search-o1
  • 报告生成任务中也显著优于基线方法

局限性

▼
  • 论文主要基于 QwQ-32B 作为骨干模型，未在其他规模或架构的模型上充分验证通用性
  • Deep Web Explorer 的搜索深度受限于 API 调用次数和网页内容获取的质量
  • 报告生成的质量评估依赖于 LLM 评判（DeepSeek-R1 + GPT-4o），可能存在评判偏差
  • 迭代 DPO 训练的计算成本较高，每轮迭代需要重新采样轨迹
  • 模型在某些 Level 3（最难级别）的任务上仍有较大提升空间

个人思考

这篇论文的核心贡献在于将"深度搜索"和"报告生成"整合到推理模型的推理链中，而非作为外部附加模块。这种设计让搜索和写作成为了推理的自然组成部分。Deep Web Explorer 的递归设计特别值得关注——它允许模型在搜索结果中层层深入，比简单 RAG 更接近人类研究行为。WebThinker 代表了 AI 助手从"问答工具"向"研究助手"演进的重要一步。该工作被 NeurIPS 2025 接收。
