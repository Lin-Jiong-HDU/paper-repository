# Endless Terminals: Scaling RL Environments for Terminal Agents

## 基本信息

- **标题**: Endless Terminals: Scaling RL Environments for Terminal Agents
- **作者**: Kanishk Gandhi, Shivam Garg, Noah D. Goodman, Dimitris Papailiopoulos
- **发表日期**: 2026-01-23
- **arXiv ID**: 2601.16443
- **arXiv 链接**: <https://arxiv.org/abs/2601.16443>
- **领域/关键词**: 强化学习, 程序化生成, 终端代理, PPO, 任务自动化

## 一句话总结
> 通过程序化生成大规模终端任务环境（Endless Terminals，3255个任务），使用最朴素的PPO算法和二元奖励信号即可训练出具备终端操作能力的语言模型代理，在TerminalBench 2.0基准上超越了需要复杂脚手架系统的Claude Sonnet 4.5。

## 论文精读

### 背景与前置知识

理解本文需要以下基础概念：

**强化学习（RL）与PPO**：强化学习是让智能体通过与环境交互、获取奖励来学习策略的方法。PPO（Proximal Policy Optimization）是目前最常用的RL算法之一，它通过限制策略更新幅度来保证训练稳定性。在本文中，每个终端任务就是一个RL episode，模型执行一系列命令，最终根据任务是否完成获得1或0的奖励。

**语言模型代理（LLM Agent）**：指能够理解自然语言指令、并通过工具调用（如执行终端命令）来完成任务的LLM。与单纯的文本生成不同，代理需要在多轮交互中规划、执行、观察反馈并调整策略。

**程序化生成（Procedural Generation）**：通过算法自动创建内容的方法。本文用LLM自动生成任务描述，再用自动化流程将其转化为可执行的训练环境，从而绕过人工设计的瓶颈。

**终端环境与容器化**：本文的训练环境运行在Docker容器中，每个任务对应一个独立的Linux终端环境，模型通过shell命令与之交互。容器化确保了任务之间的隔离性和可重复性。

**脚手架系统（Scaffolding）**：在Agent领域，脚手架指围绕LLM构建的辅助系统，包括多Agent协作、记忆管理、工具调用编排等。本文的核心发现之一是：当环境规模足够大时，简单的RL训练可以替代复杂的脚手架。

### 核心思想详解

本文的核心洞察可以用一句话概括：**与其花精力设计复杂的训练方法，不如把精力花在扩大环境规模上**。

打个比方：教一个人做饭，与其精心设计一套包含10道菜的详细教学大纲（复杂脚手架），不如给他1000道不同的菜让他自己摸索（大规模环境）。菜谱够多、够多样，学习者自然能从中学到通用的烹饪技巧。

具体来说，作者构建了一个名为"Endless Terminals"的环境系统，通过4阶段流水线自动生成终端任务：

1. **LLM生成任务描述**：让语言模型编写各种终端操作任务的详细说明书
2. **容器化环境搭建**：根据描述自动创建Docker容器，安装所需软件包
3. **完成测试生成**：自动编写测试脚本来判断任务是否被正确完成
4. **基于解的过滤**：尝试运行参考解答，过滤掉不可解或环境有问题的任务

这个流程产生了3255个训练任务，覆盖了文件操作、文本处理、系统管理、网络操作等多种终端技能领域。

训练方法极其简单——标准的PPO算法，没有任何花哨的技巧：
- 二元奖励：任务完成给1，否则给0
- 没有过程奖励、没有课程学习、没有奖励塑形
- 每个episode最多允许30轮交互

### 方法逐步拆解

**第一步：任务描述生成**
- 使用LLM（具体为语言模型）生成终端任务的详细描述
- 每个描述包含：任务目标、所需工具、预期结果、难度等级
- 生成的任务覆盖多种领域：文件管理、数据处理、系统配置、网络操作等
- 这一步是整个流水线的起点，任务描述的多样性和质量直接影响后续所有步骤

**第二步：容器环境搭建与验证**
- 根据任务描述自动创建Docker容器
- 在容器中安装任务所需的所有软件包和依赖
- 运行验证脚本确保环境正确配置（例如，检查所需的命令行工具是否可用）
- 这一阶段的通过率约为描述生成量的某个比例，过滤掉环境搭建失败的任务

**第三步：完成测试生成**
- 自动编写shell测试脚本，用于判定任务是否被正确完成
- 测试脚本检查：文件是否存在且内容正确、进程是否运行、配置是否正确等
- 测试的准确性至关重要——它既是训练的奖励信号来源，也是质量控制的关卡

**第四步：基于解的过滤**
- 使用LLM尝试生成每个任务的参考解答
- 在容器中运行参考解答，验证其能否通过完成测试
- 过滤掉三类有问题的任务：(1) LLM无法解决的任务（可能太难或有歧义）；(2) 容器环境有bug的任务；(3) 测试脚本有误的任务
- 最终保留3255个经过验证的高质量任务

**第五步：RL训练**
- 使用标准PPO算法训练语言模型
- 状态：当前终端输出（历史命令和响应）
- 动作：下一条要执行的终端命令
- 奖励：episode结束时，任务完成得1，否则得0
- 最大30轮交互per episode
- 训练了三个不同规模的基础模型：Llama-3.2-3B、Qwen2.5-7B、Qwen3-8B-openthinker-sft

### 关键公式/算法解读

**PPO目标函数**：PPO的核心是一个裁剪目标函数，限制策略更新幅度：

L = min(r(θ) * A, clip(r(θ), 1-ε, 1+ε) * A)

其中r(θ)是新旧策略的概率比，A是优势函数估计，ε是裁剪参数（通常为0.2）。clip操作确保策略不会在单步更新中变化过大，保证训练稳定性。

**二元奖励函数**：本文使用最简单的奖励设计：

R = 1 if task_completed else 0

这个奖励只在episode结束时给出，中间步骤没有任何奖励信号。这种设计的优势是不需要人工设计密集的奖励函数，劣势是信用分配（credit assignment）更困难——模型需要自己发现哪一步做对了。

**pass@k指标**：评估时使用pass@k来衡量模型的代码生成能力：

pass@k = 1 - (C(n-c, k)) / C(n, k)

其中n是总采样数，c是成功数，k是取的top-k。pass@1就是单次通过率，pass@5表示5次尝试中至少成功一次的概率。

### 实验设计分析

**为什么选这些数据集？**
- **Endless Terminals dev set（3255任务）**：作为训练数据，规模足够大且覆盖面广
- **TerminalBench 2.0**：作为分布外（OOD）评估基准，测试模型在未见过的任务上的泛化能力
- **OpenThinker dev set**：用于对比RL训练数据来源的影响

**为什么选这些指标？**
- **pass rate / pass@5 / pass@16**：不同粒度的成功率指标，pass@1最严格，pass@16最宽松
- 使用多个k值可以全面了解模型的可靠性——高pass@1意味着高可靠性，高pass@16但低pass@1意味着有潜力但不稳定

**基线选择的逻辑**
- **Base models**：展示RL训练的绝对提升
- **RL(OpenThinker)**：用不同来源的RL数据训练，验证Endless Terminals数据的优势
- **Terminus-2 scaffold**：在基础模型上叠加复杂的脚手架系统
- **Claude Sonnet 4.5 + Terminus-2**：用最强闭源模型+脚手架作为上限参考

**结果说明什么？**
- Endless Terminals RL训练在所有模型上都有显著提升，即使是最小的3B模型也提升了14个百分点
- 在TerminalBench 2.0上的正向迁移证明了方法学到了可泛化的终端操作技能，而非仅仅是记忆训练任务
- Qwen3模型在TerminalBench 2.0上甚至超越了Claude Sonnet 4.5+脚手架的组合，说明简单RL+大规模环境的威力

## 问题与动机

现有的终端Agent训练面临一个根本矛盾：**高质量终端任务环境的稀缺性**。人工设计每个训练任务（包括环境配置、测试编写、答案验证）极其耗时，这导致现有的终端Agent训练环境规模有限。

同时，当前的SOTA终端Agent（如Claude Sonnet 4.5配合Terminus脚手架）依赖复杂的系统工程——多Agent系统、精心设计的提示链、复杂的记忆管理等。这种复杂性的必要性并不明确：是因为模型能力不足才需要这么多辅助，还是这些辅助本身就是必需的？

本文提出的假设是：**当环境规模足够大时，简单的RL训练就够了**。通过程序化生成大量训练环境，可以让模型从多样的任务经验中自然习得通用能力，而无需依赖复杂的脚手架系统。

## 核心方法

核心方法包含两个紧密联系的部分：

1. **Endless Terminals环境生成流水线**：一个自动化的4阶段流程，从LLM生成任务描述开始，经过容器搭建、测试生成、基于解的过滤，最终产出高质量的可训练任务。关键创新在于将环境创建自动化，使得训练数据规模可以轻松扩展。

2. **Vanilla PPO训练**：使用最标准的PPO配置进行训练。不做任何特殊优化——没有课程学习、没有过程奖励、没有多轮对话管理。这种极简设计是有意为之，目的是证明"环境规模>算法复杂度"的假设。

## 实验结果

- **数据集**: Endless Terminals (3255个程序化生成任务), TerminalBench 2.0 (OOD评估), OpenThinker dev set
- **主要指标**: pass rate, pass@5, pass@16
- **关键结果**:
  - Dev set提升: Llama-3.2-3B 4.0%→18.2%, Qwen2.5-7B 10.7%→53.3%, Qwen3-8B 42.6%→59.0%
  - TerminalBench 2.0迁移: Llama 0.0%→2.2%, Qwen 2.2%→3.4%, Qwen3 1.1%→6.7%
  - Qwen3+RL在TerminalBench 2.0上超越Claude Sonnet 4.5+Terminus-2
- **对比方法**: Base models, RL(OpenThinker), Terminus-2 scaffold, Claude Sonnet 4.5+Terminus-2

## 局限性

1. **规模天花板**：虽然3255个任务已经比现有工作多很多，但与代码生成领域的预训练数据规模相比仍然很小。更大的环境规模是否还能持续带来改进，目前未知。

2. **任务多样性受限**：程序化生成的任务天然倾向于模型（LLM）擅长描述的模式，可能存在系统性盲区——某些类型的终端任务很难通过LLM生成出来。

3. **奖励信号稀疏**：二元episode级奖励在长episode中的信用分配问题仍然严重。30轮交互中只有最后一步有信号，中间过程的优化完全依赖模型自己摸索。

4. **评估基准局限**：TerminalBench 2.0作为OOD评估基准，其任务分布可能与Endless Terminals的训练分布仍有一定重叠（都是终端操作任务），真正的泛化能力可能需要更异构的评估来验证。

5. **计算成本**：虽然算法简单，但在3255个任务上运行PPO训练（每个任务多次rollout、容器化环境）的计算开销仍然不小。

## 个人思考

本文最有价值的贡献不在于具体的数值提升，而在于提供了一个清晰的实验证据：**在Agent训练中，环境规模可能比算法复杂度更重要**。这与"苦涩教训"(Bitter Lesson)的思想一脉相承——简单方法+大规模数据，往往胜过精心设计的小规模方法。

程序化生成环境的4阶段流水线设计很实用，特别是"基于解的过滤"这一步——它巧妙地将环境质量控制和训练数据筛选统一在同一个流程中。不过，这种依赖LLM生成+LLM验证的闭环可能存在"盲点"：如果生成器和验证器共享相同的系统性偏差，一些有问题的任务可能无法被过滤掉。

从实际应用角度看，本文的方法论具有很强的可迁移性——同样的流水线可以用于生成其他类型Agent的训练环境（如浏览器操作、数据库查询等）。核心思路是：自动化创建环境→大规模训练→简单算法即可。
