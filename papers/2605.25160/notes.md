# SimuWoB: 模拟真实移动应用用于快速且忠实的GUI Agent基准测试

## 基本信息

- **标题**: SimuWoB: Simulating Real-World Mobile Apps for Fast and Faithful GUI Agent Benchmarking
- **作者**: Guohong Liu, Jialei Ye, Pengzhi Gao, Wei Liu, Jian Luan, Yunxin Liu, Yuanchun Li (清华大学AIR, 电子科技大学, 小米)
- **发表日期**: 2026年5月
- **arXiv ID**: 2605.25160
- **arXiv 链接**: <https://arxiv.org/abs/2605.25160>
- **领域/关键词**: GUI agents, benchmarking, 移动应用仿真, 合成基准, 虚拟环境

## 一句话总结
>
> SimuWoB 提出了一种基于LLM的移动应用环境合成框架，自动生成120个高保真、带可执行验证器的可交互移动应用测试环境，揭示当前最强移动GUI agent在复杂任务上的成功率仅27.92%。

## 论文精读

### 背景与前置知识

**移动GUI Agent**：能够像人类一样操作手机应用的AI系统。给定一个任务（如"帮我买双鞋"），agent可以观察屏幕、推理当前状态、执行点击/滑动等操作，直到任务完成。

**现有基准测试的三大问题**：

1. **真实感与多样性不足**：为保持可复现性，许多基准只能用开源应用或文件操作任务，导致与实际使用场景存在差距。比如测试"在淘宝上搜索商品并下单"时，由于无法连接真实服务器，只能测试"打开设置应用"这类简单任务。

2. **任务复杂度有限**：很多基准主要测试视觉定位（点对位置）和简单导航，缺乏对长周期执行、中间信息管理和多步推理的压力测试。

3. **评估效率低**：许多交互式基准依赖模拟器、虚拟机等，启动缓慢、状态重置复杂，限制了大规模评估和在线强化学习训练。

**关键概念**：
- **无后端网页（backend-free webpage）**：所有数据和逻辑都在前端（浏览器端），无需服务器，可独立运行的网页应用
- **奖励函数/验证器**：判断任务是否完成的自动检查条件
- **成功率（Success Rate, SR）**：agent成功完成任务的比例

### 核心思想详解

**一个直觉类比**：想测试一个实习生能否熟练使用各种手机App。传统方法是用真实App——但问题是你得准备测试账号、清空购物车、重置数据库，而且不知道实习生有没有真的完成任务。另一种方法是先让实习生看题目截图——但这测不出实际操作能力。

SimuWoB 的做法是：**用AI生成"仿真的App"**。这些App看起来和真实App一模一样（UI风格、交互逻辑），但所有数据都是假的、模拟的，并且在浏览器中就能运行，不需要安装App或连接服务器。

更巧妙的是，因为SimuWoB自己"制造"了这些App，它们知道内部状态，所以可以精确地判断任务是否完成——比如"检查用户是否把3件商品加入购物车，且总价不超过预算"——这是真实App无法做到的。

**两个层次的关键创新**：

1. **环境生成自动化**：用LLM代码生成能力，从自然语言任务描述出发，自动迭代生成一个完整的移动应用仿真，包括UI布局、交互逻辑、数据实体、任务验证器。

2. **环境与任务解耦**：先生成App本身（第一阶段），再注入具体任务和验证器（第二阶段）。这种解耦确保了生成的App不是只为一个任务"量身定制"的，而是真正可用的多功能应用。

### 方法逐步拆解

**Stage 1: 最小工作环境（MWE）构建**

1. 收集应用元数据：名称、视觉风格、功能摘要、核心交互逻辑。有截图时也提供截图。
2. 代码生成LLM（如Gemini或Claude）进行迭代构建循环：
   - 起草或更新PRD（产品需求文档），设计应用的页面、功能、设计风格
   - 实现或修改页面结构、数据模式和交互逻辑
   - 进行自我审查，检查完整性和一致性
   - 审查反馈回到PRD，驱动下一轮实现
3. 经过预设轮数的迭代后，获得一个稳定的MWE

**Stage 2: 任务注入和奖励合成**

1. 用更丰富的模拟内容（文本、图片、结构化记录）扩展数据库
2. 提供任务说明（包括执行意图和验证标准）
3. 任务注入agent扫描代码库，修补任务相关逻辑，合成为可执行验证器
4. 为减少对单一轨迹的过拟合，在同一App上下文中共同生成多个相关任务

**Stage 2.5: 人工辅助问题修复**

由于LLM生成复杂环境可能存在问题（交互逻辑缺陷、UI设计问题），设计了人工辅助检测修复机制：
1. 对每个生成的App bundle，验证agent交互式执行每个任务
2. 成功任务暂接受，失败轨迹发送给人工专家分类
3. 如为环境缺陷，专家提供反馈，bundle返回生成管线修复
4. 通过循环的bundle进入候选基准池，最后人工随机抽样检查

**任务分类**：

根据难度来源，SimuWoB的120个任务分为三类：
- **简单任务**（Simple）：基本导航和操作，平均约15步
- **长周期任务**（Long-Horizon）：需要长步骤链/循环，或涉及更多信息，平均约25步
- **数学相关任务**（Math-Related）：信息聚合和计算，平均约11步

**评估接口**：

每个环境暴露轻量级DOM级API：
- `window.getTasks`：返回任务对象（描述、ID、可选的JSON返回模式）
- `window.evaluateTask`：评估当前环境状态是否满足目标任务
- `window.reset`：将环境重置到初始状态

### 关键公式/算法解读

SimuWoB本身不是数学密集型工作，但有几个设计值得关注：

**任务奖励函数设计**：
- 对不需要返回值的任务：通过检查环境状态判断（如"购物车中有3件商品"）
- 对需要返回值的任务：除状态检查外，还要求agent输出符合JSON Schema的结构化数据（如{"price": 130.2}）

**环境生成的自迭代优化**：
```
循环: LLM起草PRD -> 实现代码 -> 自我审查 -> 反馈更新PRD -> 重新实现
```
这种"写PRD->写代码->审阅->改PRD->重写"的自迭代优化是保证环境质量的关键。

### 实验设计分析

**实验设置**：
- 评估5个移动GUI agent：UI-TARS-1.5、doubao-seed（豆包）、Gemini 3 Pro、MAI-UI、Mobile-Agent-v3.5
- 其中doubao-seed/TARS和Gemini为API模型，MAI-UI和GUI-Owl为开源微调模型
- 8个并行worker运行，每任务上限100步

**任务构成**：
- 63个模拟App，覆盖33个Google Play类别中的20个（超过60%）
- 涵盖中英文两种语言
- 部分任务需要返回JSON结构化结果
- 来源：86名参与者来自77个行业的260条真实需求

**关键结果**：

1. **整体表现**：平均SR仅27.92%，最佳模型（doubao-seed）也仅42.50%
2. **长周期任务**：SR骤降至17.82%（平均），显示状态管理是核心瓶颈
3. **返回格式任务**：SR显著下降（如doubao-seed从50.00%降至30.43%）
4. **真实应用验证**：在20个真实App样本任务上的测试趋势与SimuWoB一致，验证了合成环境的泛化性
5. **人类基线**：人类平均SR 92.08%，说明当前agent与人类差距巨大

**失败模式分析**：

1. **长周期状态管理失败**：agent能正确执行UI操作，但不知道在上下文之外保存中间结果。例如需要遍历历史行程累加费用时，agent翻开一张发票、查看数据、翻下一张——但忘记之前的数据。
2. **主动探索不足**：当初步搜索失败时，agent不会系统性地假设转换和探索其他入口。例如搜索邮件找不到"Client Sync"事件后，不知道切换到日历视图。
3. **细粒度控制薄弱**：滑块拖动、日期选择器、长按等精细操作，agent准确度不足。在11个细粒度控制任务中，最佳模型（Gemini 3 Pro）也只完成5/11，其他模型平均不到1.5。

## 问题与动机

移动GUI agent发展迅速，但现有基准测试在真实感、任务复杂度和评估效率三方面存有不足。真实App无法提供自动奖励函数，开源App和应用场景有限，模拟器部署开销大。需要一个既能反映真实使用场景、又能自动验证任务完成的基准测试平台。

## 核心方法

提出SimuWoB，基于LLM驱动的两阶段环境合成框架：(1) 从应用元数据和任务描述出发，LLM通过"写PRD->实现->自审"循环生成最小工作环境；(2) 注入具体任务和可执行验证器。环境为无后端网页，可通过URL访问，支持高效并行评估。共120个任务、63个模拟App，覆盖20个Google Play类别。

## 实验结果

- **数据集**: SimuWoB基准（120个任务，63个模拟App）+ 20个真实App样本任务
- **主要指标**: 成功率（SR），按任务类别（简单/长周期/数学）细分
- **关键结果**: 平均SR 27.92%，人类92.08%；长周期任务SR 17.82%；合成环境评估趋势与真实环境一致
- **对比方法**: UI-TARS-1.5, doubao-seed, Gemini 3 Pro, MAI-UI, GUI-Owl-1.5

## 局限性

1. 仅支持网页基的观察（截图），不支持Accessibility Trees等结构化信号
2. 仅覆盖单App工作流，不支持跨App任务
3. 当前无法评估需要权限（相机、定位）的应用
4. 应用UI可能不完全复现真实App的所有交互细节
5. 环境生成依赖LLM，构建过程需要人工审查环节

## 个人思考

SimuWoB在GUI agent评测方面做出了实质性的贡献。最有价值的是其"用合成环境替代真实环境"的评测理念——解决了长期困扰该领域的奖励函数自动化和环境复现性问题。

几个值得关注的亮点：
- 环境生成中"PRD->实现->自审"的自迭代优化思路，类似于人类"先设计再实现再审查"的工作流程，有效提高了生成质量
- 人工辅助的问题修复机制体现了"人机协同"的实用主义——承认LLM生成的局限性，但不放弃自动化
- 失败模式分析深入且有实际指导意义——特别是"长周期状态管理"和"主动探索不足"的发现，为后续agent改进指明了方向

与已有基准（如AndroidWorld）的对比显示SimuWoB更具挑战性——平均SR 27.92% vs 69.38%，这说明简单操作能力已经不再是瓶颈，而复杂推理和状态管理才是。
