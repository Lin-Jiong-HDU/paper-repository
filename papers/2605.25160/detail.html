SimuWoB: 模拟真实移动应用用于快速且忠实的GUI Agent基准测试

Guohong Liu, Jialei Ye, Pengzhi Gao et al. (清华大学AIR, 电子科技大学, 小米) 2026年5月
arXiv:2605.25160
GUI Agents Mobile Apps Benchmarking Synthetic Environment LLM

一句话总结

SimuWoB 提出基于LLM的移动应用环境合成框架，自动生成120个高保真、带可执行验证器的可交互移动应用测试环境，揭示当前最强移动GUI agent在复杂任务上成功率仅27.92%。

背景与前置知识

▼

移动GUI Agent

能像人类一样操作手机应用的AI系统。给定任务后，agent可以观察屏幕、推理当前状态、执行操作直到任务完成。

现有基准测试三大问题

  • 真实感与多样性不足：为保持可复现性，只能用开源应用或文件操作任务，与实际使用场景有差距
  • 任务复杂度有限：主要测试视觉定位和简单导航，缺乏对长周期执行、中间信息管理、多步推理的压力测试
  • 评估效率低：依赖模拟器、虚拟机，启动缓慢、状态重置复杂

关键概念

无后端网页：所有数据和逻辑在前端，无需服务器。因为SimuWoB"制造"了这些App，它们知道内部状态，可以精确判断任务是否完成——这是真实App无法做到的。

核心思想详解

▼

想测试实习生能否熟练使用各种手机App。用真实App需要准备账号、清理数据、且无法自动判断是否完成任务。SimuWoB的做法：用AI生成"仿真的App"——看起来和真实App一模一样，但数据和逻辑都是模拟的。

两个层次的关键创新

  • 环境生成自动化：用LLM代码生成能力，从自然语言任务描述出发，自动生成完整的移动应用仿真
  • 环境与任务解耦：先生成App本身，再注入具体任务和验证器，确保App是真正可用的多功能应用而非为单一任务定制

方法逐步拆解

▼

Stage 1: 最小工作环境构建

LLM进行迭代构建循环：起草PRD -> 实现页面和交互 -> 自我审查 -> 反馈更新PRD -> 重新实现。获得稳定的最小工作环境。

Stage 2: 任务注入和奖励合成

扩展数据库 -> 提供任务说明 -> Agent扫描代码库 -> 合成可执行验证器。因为控制环境状态，验证器可以精确检查成功条件。

人工辅助修复

验证agent执行每个任务 -> 成功接受/失败人工分类 -> 环境缺陷返回修复 -> 人工随机抽样检查确保质量。

任务分类

  • 简单任务（Simple）：基本导航和操作，~15步
  • 长周期任务（Long-Horizon）：长步骤链/循环，~25步
  • 数学相关任务（Math-Related）：信息聚合和计算，~11步

实验分析

▼

评估设置

5个移动GUI agent（UI-TARS-1.5, doubao-seed, Gemini 3 Pro, MAI-UI, GUI-Owl-1.5），8并行worker，每任务上限100步。

关键结果

  • 平均SR仅27.92%，最佳（doubao-seed）42.50%
  • 长周期任务SR仅17.82%
  • 人类基线92.08%——差距巨大
  • 合成环境趋势与真实环境一致

三种失败模式

  • 状态管理失败：能正确操作UI但不知道保存中间结果
  • 主动探索不足：搜索失败后不会系统性切换策略
  • 细粒度控制薄弱：滑块拖动、日期选择等精细操作准确度低

局限性

▼
  • 仅支持网页基观察（截图），不支持Accessibility Trees
  • 仅覆盖单App工作流，不支持跨App任务
  • 无法评估需要权限（相机、定位）的应用
  • 应用UI可能不完全复现真实App所有细节
  • 环境生成依赖LLM，需要人工审查

个人思考

SimuWoB最有价值的是"用合成环境替代真实环境"的评测理念，解决了奖励函数自动化和环境复现性问题。环境生成中"PRD->实现->自审"的自迭代优化思路，类似人类"先设计再实现再审查"的工作流程。人工辅助修复机制体现了实用主义。失败模式分析深入——"长周期状态管理"和"主动探索不足"的发现为后续agent改进指明方向。与AndroidWorld对比（SR 27.92% vs 69.38%）说明简单操作不再是瓶颈，复杂推理和状态管理才是。
