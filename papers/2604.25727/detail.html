Towards Scalable Terminal Task Synthesis via Skill Graphs

Zhiyuan Fan, Tinghao Yu, Yuanjun Cai, Jiangtao Guan, Yun Yang, Dingxin Hu, Jiang Zhou, Xing Wu, Zhuo Han, Feng Zhang, Lilin Wang 2026年4月28日
arXiv:2604.25727
Terminal Agents Skill Graph Task Synthesis Synthetic Data Multi-Agent Systems CLI Automation

一句话总结

SkillSynth 通过构建场景中介的技能图谱，从图中采样路径并利用多智能体系统自动合成多样化的终端任务实例，从而为训练终端智能体提供高质量、可扩展的执行轨迹数据。

背景与前置知识

▼

终端智能体（Terminal Agents）

大型语言模型通过命令行界面（CLI）与计算系统交互，自主执行复杂任务。这类智能体不依赖专有 API，而是将终端作为通用的动作空间。核心挑战在于：如何获取足够多样、高质量的训练轨迹？

场景（Scenario）与技能（Skill）

终端智能体的执行轨迹可以抽象为"场景-技能"的交替序列。场景描述环境在某个决策点的语义状态（如"含有失败单元测试的 Python 项目"），技能是在该场景下执行的一系列连贯动作（如"运行 pytest 并捕获输出"），将环境从当前场景转移到下一个场景。

技能图谱（Skill Graph）

以场景为节点、技能为有向边构建的图结构。图中的有向路径对应一个真实的工作流程——技能的后置条件恰好是下一个技能的前置条件。

层次强化学习中的 Options

论文中的场景-技能抽象借鉴了层次强化学习中 options 的概念，将低级的动作序列提升到高级的语义层面来分析和优化。

核心思想详解

▼

SkillSynth 的核心洞察是：训练终端智能体的关键不只是任务数量，而是执行轨迹在场景和技能两个维度上的多样性。

用一个类比来理解：想象一个厨师学徒的训练。传统方法只是不断增加菜谱数量（更多任务实例），但学徒反复练习的烹饪场景和手法可能高度重叠。SkillSynth 的做法是先构建一张"烹饪技能图谱"——节点是厨房中的各种状态（如"准备好的食材"、"半成品"），边是烹饪技能（如"切配"、"翻炒"），然后从图中采样路径来设计训练菜单，确保学徒在多样的厨房状态中练习多样的烹饪技能。

这个框架有三层结构：

  • 技能图谱构建：从 ClawHub 和 GitHub 收集真实终端技能，用 LLM 推断每个技能的前置/后置场景，通过聚类去重和跨技能对齐，构建出包含 82,073 个场景节点和 57,214 条技能边的有向图。
  • 图引导路径采样：使用逆频率加权采样算法，从图中采样有向路径作为工作流程的抽象。逆频率策略避免了高频节点和边的过度覆盖。
  • 多智能体合成：将采样的路径通过 Planner-Constructor 分工的多智能体系统实例化为可执行的任务，双重验证确保任务可解且质量达标。

方法逐步拆解

▼

第一阶段：技能图谱构建

  1. 技能过滤：从 ClawHub 和公开 GitHub 仓库收集原始技能，保留满足四个条件的技能——可在 Linux 终端执行、有结构化工作流定义、无恶意内容、产生确定性可验证输出。
  2. 场景推断：对每个保留的技能，用 LLM 推断其前置场景和后置场景，形成原子转移。
  3. 场景去重：将所有推断的场景用 embedding 进行聚类去重，合并描述同一状态但措辞不同的场景。通过实验比较了九种聚类算法，发现基于 Louvain 粗分桶的层次聚合聚类效果最好。
  4. 跨技能对齐：将一个技能的后置条件与另一个技能的前置条件对齐——通过 embedding 相似度检索 top-1000 候选，再用 LLM 判断语义兼容性。
  5. 合并与过滤：将对齐的前置-后置对合并为统一场景节点，最后用 LLM 过滤所有三元组，保留有效转移。

第二阶段：图引导路径采样

  1. 逆频率采样：维护每个场景的访问计数和每个技能的使用计数，起始场景和每一步的边选择都以逆频率为权重，偏向尚未被充分覆盖的节点和边。
  2. 单调性约束：在同一条路径内，已访问的场景和技能不再重复，确保路径单调递进。
  3. 路径过滤：只接受长度在 [1, 7] 范围内且技能组合未出现过的新路径。

第三阶段：多智能体合成

  1. Planner 规划：将采样路径转化为结构化的子目标和预期输出计划。
  2. Constructor 构建：根据计划生成完整的任务实例（指令、文件系统、Docker 环境、测试脚本、Oracle 解）。
  3. 双重验证：执行验证 + 规则评估（LLM-as-a-Judge）。
  4. 修复循环：验证失败时返回诊断信息修复，最多 3 轮、每轮最多 20 次工具调用。

关键公式/算法解读

▼

场景-技能抽象（公式 1-2）

κ_t: σ_{t-1} → σ_t 表示技能 κ_t 将环境从场景 σ_{t-1} 转移到 σ_t。一条完整的执行轨迹 ξ = (σ_0, κ_1, σ_1, ..., κ_L, σ_L) 是场景和技能的交替序列。

技能级训练目标（公式 3）

J(π) = E_{ξ~D} [Σ log π(κ_t | σ_{t-1}, g)]，即在训练轨迹的经验分布 D 下，最大化智能体在每个场景下选择正确技能的对数似然。等价于标准 next-token prediction loss。

覆盖准则分解（公式 4）

将目标分解为场景分布和条件技能分布的乘积。关键洞察：智能体只能学到 D 支撑集内的场景和技能，因此训练数据需要密集覆盖 {(σ, κ)} 的条件乘积空间。这直接 motivating 了通过最大化轨迹多样性来提升学习能力的思路。

逆频率路径采样（Algorithm 1）

核心思想是用逆频率权重偏向采样尚未充分覆盖的场景和技能，逐步将经验分布推向均匀覆盖。结合单调性约束和技能组合去重，确保采样路径的多样性。

实验设计分析

▼

为什么选 Terminal-Bench？ 它是目前终端智能体最权威的评测基准，1.0 有 80 个任务，2.0 有 89 个更难的任务，要求在 Docker 容器中完成端到端工作流。

为什么用 Qwen3 系列？ 选择 8B、14B、32B 三个规模研究模型规模效应，Qwen3-32B 作为主要消融实验模型。

关键发现：

  • 单次全自动运行合成 3,560 个可用任务，Oracle 通过率 95.7%，平均成本仅 $27.3/任务。
  • Claude Opus 4.6 平均需要 37 步来解，121 个任务在三次运行中无法解决。
  • Qwen3-32B + SkillSynth 在 TB 2.0 上超越更大的 Qwen 3 Coder 480B，说明领域特定数据比模型规模更重要。
  • 相比单技能和随机多技能基线，场景-技能对覆盖率分别高 31% 和 19%。

局限性

▼
  • 依赖 LLM 进行场景推断、对齐和质量评估，成本和 LLM 本身的不确定性会影响图谱质量。
  • 路径采样仅考虑链式结构（线性序列），不支持子图采样（技能并行执行），限制了更复杂任务的合成。
  • 首轮生成质量差的任务难以通过修复循环恢复，主要瓶颈是损坏的文件系统快照。
  • 评测主要集中在 Terminal-Bench，在其他基准上的泛化性尚未验证。
  • 训练轨迹收集依赖特定教师模型，不同教师模型的效果未探索。

个人思考

将终端任务多样性形式化为场景-技能对覆盖问题，用图结构实现可控采样，是很有创意的做法。场景中介的设计让技能之间有了语义桥梁。端到端自动化程度高（95.7% 通过率、$27.3/任务），对工业界很有吸引力。如果能扩展到子图采样和增量更新，以及迁移到浏览器/代码编辑智能体，价值会更大。
