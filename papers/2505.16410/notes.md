# Tool-Star: Empowering LLM-Brained Multi-Tool Reasoner via Reinforcement Learning

## 基本信息

- **标题**: Tool-Star: Empowering LLM-Brained Multi-Tool Reasoner via Reinforcement Learning
- **作者**: Guanting Dong, Yifei Chen, Xiaoxi Li, Jiajie Jin, Hongjin Qian, Yutao Zhu, Hangyu Mao, Guorui Zhou, Zhicheng Dou, Ji-Rong Wen
- **发表日期**: 2025-05-22
- **arXiv ID**: 2505.16410
- **arXiv 链接**: <https://arxiv.org/abs/2505.16410>
- **领域/关键词**: 强化学习, 工具集成推理, 多工具协作, LLM 推理, GRPO, 数据合成

## 一句话总结
>
> Tool-Star 是一个基于强化学习的框架，通过两阶段训练（冷启动微调 + 多工具自批评 RL）和数据合成管线，使 LLM 能够在逐步推理过程中自主调用搜索引擎、代码解释器等多种外部工具，在 10+ 个挑战性推理基准上显著超越了单工具 RL 方法。

## 论文精读

### 背景与前置知识

**工具集成推理 (Tool-Integrated Reasoning, TIR)**：传统 LLM 只靠语言本身进行推理，而 TIR 让模型在推理链中嵌入对外部工具（如搜索引擎、代码执行器）的调用，把"纯语言推理"扩展为"交互式推理"。类比：如果 LLM 的推理像闭卷考试，TIR 就像开卷考试 + 可以用计算器 + 可以查资料。

**强化学习 (RL) 与 GRPO**：RL 通过奖励信号优化模型行为。GRPO (Group Relative Policy Optimization) 是 DeepSeekMath 提出的 RL 算法，用一组 rollout 的相对表现来估计 baseline，无需单独训练 critic 模型。类比：GRPO 像是"小组成员互评"，每次生成一组答案，组内相互比较来决定谁好谁坏。

**课程学习 (Curriculum Learning)**：先学简单的，再学难的，循序渐进地提升模型能力。Tool-Star 将数据按难度分为简单/困难两组，分别用于 SFT 和 RL 阶段。

**SFT 与 DPO**：SFT（监督微调）让模型模仿好的示例；DPO（直接偏好优化）通过对比"好答案"和"坏答案"来训练模型偏好。

### 核心思想详解

Tool-Star 要解决的核心问题是：**如何让 LLM 在推理时不仅能用一个工具，而是能协调多个工具（搜索 + 代码 + 浏览器）进行协作推理？**

之前的 TIR 方法存在三个关键瓶颈：
1. **数据稀缺**：高质量的"带工具调用的推理链"数据很难获取
2. **单工具局限**：现有 RL 方法大多只优化单一工具（要么只用搜索，要么只用代码），导致跨领域能力不均衡
3. **奖励设计不足**：简单的"对/错"奖励无法引导模型学会多工具协作

Tool-Star 的解决思路像培养一个"全能型学生"：先通过大量练习（数据合成 + SFT）建立基本功，再通过自主探索 + 自我批评（self-critic RL）学会灵活组合不同工具。

### 方法逐步拆解

**第一步：数据合成管线（3 步）**

1. **数据收集与采样**：从开源数据集中收集约 90K 文本推理数据和 1K 已有 TIR 数据。然后通过两种方式扩展工具调用轨迹：
   - **TIR 提示采样**：用带工具调用指令的 prompt 让 LLM 生成带工具调用的推理链，自动调用工具获取反馈后拼回推理链继续生成
   - **Hint 采样**：先让模型做纯语言推理，然后在不确定的地方（"maybe"、"wait"）或答案后插入工具调用提示，让模型从该点继续进行工具增强推理

2. **工具使用质量归一化**：通过三种策略过滤低质量样本：
   - 控制工具调用频率（去除过度调用的样本）
   - 去除重复工具调用（如同一段代码反复出现）
   - 格式标准化（统一特殊 token 格式）

3. **难度感知分类**：对每个问题同时做纯语言推理和工具推理，按结果分为四类：
   - 类别 1-2：纯推理就对，不需要工具 → 进入 SFT 的文本子集
   - 类别 3：工具推理才对，证明工具有效 → 进入 SFT 的工具子集
   - 类别 4：两种都不对，最难 → 专门留给 RL 阶段

**第二步：两阶段训练框架**

**阶段 1 - 冷启动 SFT**：用阶段 1 的数据（简单 + 中等难度）做监督微调，让模型学会基本的工具调用模式。

**阶段 2 - 多工具自批评 RL**：
1. 先用 K 步标准 GRPO RL 训练，获得 RL 初始化模型
2. 然后**自采样**：让模型自己生成多个候选答案
3. 用分层奖励函数给每个答案打分
4. 把高分答案作为"正例"、低分答案作为"负例"，用 DPO 做偏好学习
5. 这个自批评过程每 K 步穿插一次，迭代进行

**六种工具设计**：
- 训练工具：搜索引擎、Web 浏览器代理、代码解释器
- 推理工具：代码调试器（修 bug）、工具回溯器（回退到失败调用前）、推理链精炼器（超长时压缩）

### 关键公式/算法解读

**公式 (1) - 问题建模**：将多工具推理建模为自回归生成过程。推理链中每个 token 的生成概率依赖于之前所有 token、指令、问题以及之前所有工具反馈。最终答案也以自回归方式基于完整推理链生成。

**公式 (3) - 分层奖励函数**：
- 如果格式正确且答案正确：奖励 = 正确性分 + 多工具协作奖励 rM
- 如果格式正确但答案错误：奖励 = 0
- 格式错误：奖励 = -1
- rM（多工具协作奖励）：当推理链中同时出现 `<search>` 和 `<python>` 时给 0.1 额外奖励，否则为 0
- 这意味着同时使用搜索和代码会得到更高奖励

**公式 (4) - GRPO 损失**：标准 GRPO 目标，包含裁剪（clip）机制和 KL 散度惩罚。用组内 rollout 的相对优势来调整策略。

**公式 (5) - DPO 损失**：在自批评阶段，用偏好数据做 DPO 训练，让模型给高奖励答案更高的概率。

### 实验设计分析

**数据集选择**：覆盖两大类共 10+ 个数据集：
- 数学推理：AIME24、AIME25、MATH500、MATH、GSM8K（由易到难，从基础算术到竞赛数学）
- 知识推理：WebWalker、HotpotQA、2WikiMultiHopQA、MuSiQue、Bamboogle（多跳 QA，测试信息检索和整合能力）

这样设计是为了验证模型在**计算型**和**知识型**两类核心推理能力上的表现。

**基线方法**：覆盖三大类方法：
- 基座模型（Qwen2.5-3B、Llama3.2-3B）
- 单工具增强（ToRL 代码、DotaMath 代码、RAG/Search-o1/Search-R1 搜索）
- 多工具方法（Multi-Tool Prompting、ReCall）

**关键发现**：
1. 仅靠 prompt 引导的多工具使用反而不如基座模型（不稳定）
2. 单工具 RL 方法在擅长领域很强，但在其他领域表现急剧下降（如搜索方法在 MATH500 上准确率暴跌 20%+）
3. Tool-Star 在所有数据集上都保持强势，平均准确率 46.1%（3B 模型），显著优于所有基线

## 问题与动机

核心问题：如何让 LLM 在推理时高效、合理地自主调用多个外部工具？

两大挑战：
1. **工具使用的合理性与效率**：什么时候该用工具？用哪个？用几次？
2. **多工具协作推理**：如何让模型整合搜索引擎（获取信息）、代码解释器（精确计算）、浏览器（深度网页理解）等不同工具的能力？

## 核心方法

Tool-Star 的核心贡献是一个从数据到训练的完整框架：
- **数据层面**：自动化的 TIR 数据合成管线（提示采样 + hint 采样 + 质量过滤 + 难度分类）
- **训练层面**：两阶段框架（SFT 冷启动 → 多工具自批评 RL），配合分层奖励设计

## 实验结果

- **数据集**: AIME24, AIME25, MATH500, GSM8K, MATH, WebWalker, HotpotQA, 2WikiMultiHopQA, MuSiQue, Bamboogle
- **主要指标**: 准确率（计算推理）、F1 分数（知识推理）、工具使用效率 TE
- **关键结果**: Tool-Star(3B) 平均 46.1%，比最强单工具方法 ReCall (39.1%) 高 7 个点；工具使用效率在两类任务上均保持高水平
- **对比方法**: Qwen2.5-3B-Instruct, ToRL, DotaMath, RAG, Search-o1, Search-R1, ReCall, Multi-Tool Prompting

## 局限性

1. **规模限制**：主要实验基于 3B 参数模型，更大规模模型（7B、72B）的效果有待验证
2. **工具类型**：目前仅支持搜索引擎、代码解释器和浏览器三种训练工具，其他类型工具（如数据库查询、API 调用）的整合尚未探索
3. **延迟问题**：多工具调用带来的推理延迟未做深入分析，实际部署时可能成为瓶颈
4. **奖励设计**：分层奖励函数中的阈值（如 rM=0.1）需要手动设定，不同任务的最优值可能不同
5. **数据依赖**：数据合成管线仍依赖强模型（如 Qwen2.5-72B）生成初始推理链，成本较高

## 个人思考

Tool-Star 的设计非常系统化，从数据到训练到推理都有考虑。几个值得关注的点：

1. **Hint 采样的思路很巧妙**：不是直接让模型从零开始生成工具调用，而是从已有的推理链中找到"不确定"的节点插入工具调用，这样生成的数据质量更高
2. **自批评机制是关键创新**：标准的 GRPO 只用规则奖励，但 Tool-Star 让模型自己采样并区分好坏答案，通过 DPO 内化奖励结构——这比纯规则奖励更适合多工具协作这种复杂场景
3. **冷启动的重要性被低估**：实验表明去掉冷启动 SFT 后性能下降最大（-11.7%），说明 RL 探索的起点非常重要
4. **对实际应用的启示**：Tool-Star 验证了"小模型 + 多工具"可以接近甚至超越"大模型纯推理"的效果，这对资源受限场景很有价值
