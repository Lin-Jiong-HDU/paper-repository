# OSMa-Bench++: 面向开放式的语义建图基准测试——基于prompt生成的合成场景

## 基本信息

- **标题**: OSMa-Bench++: Toward Open-Ended Benchmarking of Semantic Mapping for Manipulation with Prompt-Generated Synthetic Scenes
- **作者**: Regina Kurkova, Maxim Popov, Sergey Kolyubin (ITMO University, BE2R Lab)
- **发表日期**: 2026年5月
- **arXiv ID**: 2605.26831
- **arXiv 链接**: <https://arxiv.org/abs/2605.26831>
- **领域/关键词**: semantic mapping, benchmarking, 语义建图, 合成场景, 压力测试, 遮挡/光照

## 一句话总结
>
> OSMa-Bench++ 提出了一种基于prompt生成合成室内场景的语义建图基准扩展方案，通过可控的合成场景生成实现对物体遮挡、杂乱布局和光照变化等操作相关边缘场景的定向压力测试。

## 论文精读

### 背景与前置知识

**语义建图（Semantic Mapping）**：机器人不仅要感知环境中的物体位置，还要理解它们是什么、彼此之间的关系。语义建图将3D场景重建与语义理解结合，生成富含语义信息的环境地图。

**下游操作任务的需求**：机器人需要操纵物体时，对语义地图有更高的要求：
- 准确知道物体的存在和数量（"有三个杯子吗？"）
- 理解物体间关系（"杯子在桌子上"）
- 知道物体的可访问性（"这个杯子被挡住了吗？"）

**现有方法的局限性**：
- 开放词汇方法（OpenScene, ConceptGraphs, BBQ等）在表达能力上有显著进步
- 但评估仍然局限在固定数据集上，缺乏对操作相关边缘场景的覆盖
- 常见的边缘场景包括：杂乱小物体、部分遮挡、光照变化、受限的物体可达性

**关键概念**：
- **SceneSmith**：一个基于agent的室内场景合成工具，能从文本描述生成仿真环境
- **Habitat**：Meta开源的3D仿真平台，广泛用于具身AI研究
- **VQA（Visual Question Answering）**：视觉问答，对场景内容进行问答测试
- **Prompt-grounded evaluation**：利用已知的场景生成prompt作为语义标注的辅助来源进行评测
- **HaDaGe**：OSMa-Bench中的相机轨迹和RGB-D序列生成器

### 核心思想详解

**一个直觉类比**：要测试一个学生的空间感知能力，你不能永远只用在同一个教室里考试（固定数据集）。你需要能够灵活地布置考场——有时放很多小物体看学生能否分辨，有时调暗灯光测试夜视能力，有时把物体放到遮挡位置之后。

OSMa-Bench++ 做的就是：**根据需求自动生成各种"考场"（合成室内场景），然后在每个考场里系统性地测试机器人的语义理解能力**。

核心创新点有两个：

1. **可控的场景生成管线**：用LLM生成文本描述 -> 用SceneSmith合成场景 -> 转换格式 -> 生成观察序列 -> 评估语义地图。这个管线让研究人员可以针对性地生成特定类型的测试场景（如"很多小物体放在架子上"或"光照不均匀的房间"）。

2. **Prompt-grounded评估**：因为场景的生成prompt是已知的，它本身就可以作为语义标注的"参考答案"。这在评估物体计数和关系等依赖视角的问题时尤为有价值——传统VQA只能根据agent看到的内容问问题，但有些物体可能从某些视角看不到。有了prompt，可以问"根据场景设计，这个房间里应该有几个杯子？"而不是"你在这个视角看到了几个杯子？"

### 方法逐步拆解

**Step 1: Prompt生成和过滤**

1. 用LLM生成一批候选场景描述prompt
2. 用text embedding模型（如text-embedding-3）将prompt编码为向量
3. 计算prompt之间的余弦相似度，移除近似重复
4. 从剩余pool中选择最多样化的子集（选择与已选prompt距离最远的）

**Step 2: 场景合成（SceneSmith）**

SceneSmith根据每个prompt生成完整的室内3D场景，包括家具布局、物体摆放、材质纹理等。输出为Drake仿真格式。

**Step 3: 格式转换和适配**

关键：SceneSmith输出Drake格式 -> 需要转换为Habitat兼容格式
适配工作包括：
- 语义标签归一化（不同工具用的类别名称可能不同）
- 材质和纹理修复
- 着色器降级策略（当高级着色器不兼容时使用替代方案）
- 地板处理
- 导航网格设置
- 可控光照配置（四种光照条件）

**Step 4: 观察序列生成**

使用HaDaGe生成器生成RGB-D观察序列和相机轨迹。

**Step 5: 语义建图方法运行**

在生成的序列上运行被评估的语义建图方法（ConceptGraphs和BBQ），生成3D地图和场景图。

**Step 6: 评估**

两个评估维度：

1. **分割质量评估**：mAcc（平均准确率）和 f-mIoU（频率加权平均交并比）
2. **场景图完整性评估**：通过VQA测试场景图的完整性和准确性

**Prompt-Grounded VQA**：
- 传统VQA问题只基于agent在轨迹中看到的内容
- Prompt-grounded VQA利用原始生成prompt作为额外标注来源
- 重点关注两类问题：
  - **测量类（Measurements）**：物体数量是否正确
  - **关系类（Relations between objects）**：物体间空间关系是否正确

### 关键公式/算法解读

**Prompt多样性选择**：

$$\text{sim}(p_i, p_j) = \frac{f(p_i)^\top f(p_j)}{\|f(p_i)\|_2 \|f(p_j)\|_2}$$

- $p_i$ 和 $p_j$ 是两个场景描述prompt
- $f(\cdot)$ 是文本嵌入模型
- 计算余弦相似度来检测重复和选择多样化prompt

这个公式简单但有效——在自动生成场景描述时，保证场景多样性的一种实用方法。

**四种光照条件**：
- **Baseline**：场景原生提供的静态非均匀分布式光源
- **Nominal Lights**：移除所有显式光源，仅依赖mesh自发光
- **Camera Light**：在相机上附加定向光源，产生强烈的视角相关阴影
- **Dynamic Lights**：沿机器人轨迹变化照明，模拟不同光照区域间的真实过渡

### 实验设计分析

**场景构建**：
- 初始pool：350个prompt
- 经过多样性过滤和技术验证后保留40个场景
- 分为两类：
  - **家具场景（Furniture）**：24个，评估房间级结构、物体布局、支撑关系
  - **操作物场景（Manipuland）**：16个，添加小型操作相关物体（增加杂乱度、引入细粒度关系、包括小/遮挡物体）

**质量验证**：
- 场景级别完全匹配率：36/40 = 90.0%
- 物体级别保真度：至少98.0%（假设每场景至少5个物体）
- 仅有4个轻微合成错误（各对应一个缺失物体）

**评估方法**：
- ConceptGraphs（开放词汇3D场景图）
- BBQ（更先进的开放词汇3D场景图方法）
- 每种方法在4种光照条件下评估

**关键结果**：

1. **分割性能**：
   - Camera Light对两种方法都是最大的挑战（mAcc下降最多）
   - Dynamic Light影响较小，甚至对mAcc有轻微正面作用
   - BBQ倾向于更少但几何精确的mask（高f-mIoU但低场景图完整性）
   - ConceptGraphs恢复更多物体（高完整性但精度较低）

2. **Prompt-Grounded VQA**：
   - 两种方法在测量和关系类别上的准确率均低于30%
   - BBQ在关系和操作物子集上通常优于ConceptGraphs
   - BBQ在Dynamic Light下显著下降（因为使用单帧生成物体嵌入）
   - ConceptGraphs因多视角聚合在Dynamic Light下更稳定

3. **标准VQA vs PromptGT对比**：
   - 标准VQA得分普遍高于PromptGT
   - 这说明标准VQA受限于视角依赖的可见性偏差
   - PromptGT提供了更稳定（也更严格）的场景图完整性评估

## 问题与动机

现有语义建图基准受限于固定数据集，缺乏对操作相关边缘场景的覆盖（杂乱小物体、遮挡、光照变化等）。同时，标准VQA评估依赖于特定视角下的可视性，无法可靠评估场景图对物体存在性、数量和关系等核心语义的保持情况。需要一个可扩展的评估框架，支持定向生成具有挑战性的测试场景。

## 核心方法

提出OSMa-Bench++，基于prompt驱动合成场景的扩展方案：LLM生成场景描述 -> 嵌入过滤和多样性选择 -> SceneSmith合成3D场景 -> 转换为Habitat兼容格式 -> HaDaGe生成RGB-D序列 -> 运行语义建图方法 -> 分割指标和prompt-grounded VQA评估。利用已知的生成prompt作为辅助语义标注，实现对物体计数和关系的更稳定评估。

## 实验结果

- **数据集**: 40个SceneSmith生成的室内场景（24家具+16操作物），4种光照条件
- **主要指标**: mAcc, f-mIoU（分割）；VQA准确率（场景图）
- **关键结果**: 场景级匹配率90%；VQA准确率均低于30%；不同方法对光照变化响应不同；标准VQA高估了场景图完整性
- **对比方法**: ConceptGraphs, BBQ

## 局限性

1. 当前仅评估了两种方法（ConceptGraphs, BBQ），更多方法有待验证
2. 场景合成质量依赖SceneSmith，存在少量合成错误（如缺失物体）
3. 40个场景的规模仍然有限
4. 评估聚焦在静态场景，未涉及动态环境
5. 自动生成的prompt可能不完全覆盖操作相关的所有边缘情况
6. 光照条件设置相对简单，未涵盖极端光照变化

## 个人思考

OSMa-Bench++ 解决了语义建图评估中的一个实际问题：如何系统性地测试方法对"困难情况"的鲁棒性。其核心贡献不是提出新的建图方法，而是构建了一个可扩展的评估框架。

最有启发性的是 Prompt-Grounded VQA 的设计。直觉上，如果你知道场景中应该有什么，你就可以问agent"这些东西是否在你的地图中"，而不需要依赖某个具体视角是否看到它们。这个思路的巧妙之处在于：合成场景的"已知prompt"天然就是参考答案，无需人工标注。这在其他合成数据基准中也可以借鉴。

四种光照条件的系统测试也很有价值——它揭示了不同方法对不同条件的敏感度差异。方向性光源（Camera Light）影响最大，这与直觉一致：强方向光造成的不均匀照明会干扰视觉特征提取。

另一个值得注意的观察是 BBQ 在 Dynamic Light 下的表现下降——因为它使用单帧生成物体嵌入，所以受光照影响大；而 ConceptGraphs 的多视角聚合策略对此更鲁棒。这为改进语义建图方法提供了具体方向。
