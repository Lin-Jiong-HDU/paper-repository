Agent World Model: Infinity Synthetic Environments for Agentic Reinforcement Learning

Zhaoyang Wang, Canwen Xu, Boyi Liu, Yite Wang, Siwei Han, Zhewei Yao, Huaxiu Yao, Yuxiong He 2026年2月
arXiv:2602.10090
LLM Agent 强化学习 合成环境 工具使用 多轮交互 GRPO

一句话总结

本文提出 Agent World Model (AWM)，一个全合成环境生成管线，通过自动生成 1,000 个代码驱动、数据库支撑的交互环境，实现了大规模 agentic RL 训练，并在三个基准测试上展现了强大的分布外泛化能力。

背景与前置知识

▼

LLM Agent（大语言模型智能体）：指基于大语言模型构建的自主智能体，能够通过多轮交互调用工具（如搜索、数据库查询、API 调用等）来完成复杂任务。不同于简单的问答，Agent 需要在环境中观察、规划、执行，形成闭环。

Agentic RL（智能体强化学习）：将强化学习应用于 LLM Agent 的训练过程。Agent 通过与环境的交互获得奖励信号，从而学习更好的策略。典型方法包括 PPO、GRPO 等。

GRPO（Group Relative Policy Optimization）：一种策略优化算法，通过组内相对比较来计算优势函数，避免了需要单独训练价值模型的开销。在同一组采样中，通过比较不同轨迹的回报来确定每个动作的好坏。

Tool Use（工具使用）：LLM Agent 的核心能力之一，指模型能够生成结构化的工具调用请求（如 JSON 格式的函数调用），并处理返回的观测结果。

环境（Environment）：在 RL 框架中，环境是 Agent 交互的对象。对于工具使用的 Agent 来说，环境包括一组可调用的工具、工具背后的数据/状态、以及工具执行后返回的观测。好的环境需要提供可靠的状态转移和丰富的交互空间。

合成数据生成（Synthetic Data Generation）：利用 LLM 自动生成训练所需的数据和环境，而非依赖人工标注或真实系统采集。这种方法可以大幅扩展数据规模和多样性。

核心思想详解

▼

AWM 的核心思想可以用一个类比来理解：想象你要训练一个人学习烹饪，有三种方案：

  • 方案 A（真实环境）：让他去真实餐厅实习——成本高、效率低、场景有限。
  • 方案 B（LLM 模拟环境）：让人假装是顾客和食材——不靠谱，因为"演员"可能会出错，状态不一致。
  • 方案 C（AWM 的做法）：搭建一套自动化的模拟厨房系统——有真实的食材数据库、真实的烹饪工具（代码实现的），每次操作都会真实地改变食材状态。这套系统可以批量生成上千个不同类型的"厨房"。

AWM 的关键创新在于：用代码和数据库构建可执行的合成环境，而不是用 LLM 来模拟环境。

  • 状态一致性：数据库保证了状态转移的确定性和可靠性，不会出现"幻觉"
  • 可验证性：可以通过检查数据库状态来客观验证任务是否完成，从而设计可靠的奖励函数
  • 可扩展性：通过 LLM 辅助生成，可以大规模创建多样化的环境
  • 高效性：代码执行比 LLM 模拟快得多，也比访问真实 API 便宜

方法逐步拆解

▼

第一步：场景生成

利用 LLM 生成多样化的日常场景，涵盖旅行预订、电商购物、医疗管理、餐厅运营、教育系统、金融服务等领域，共生成 1,000 个不同场景。每个场景包含：场景名称、场景描述、涉及的角色和实体。

第二步：数据库模式设计

LLM 根据场景描述生成关系型数据库表结构、字段定义、外键关系，产出完整的 SQL schema。数据库是环境状态的"真实来源"，保证状态转移的可靠性。

第三步：数据库填充

用 LLM 生成符合场景上下文的合成数据填充数据库，如用户信息、商品列表、航班数据等，确保数据的合理性和一致性。

第四步：工具实现

LLM 生成 Python 工具代码，每个工具通过 SQL 查询与数据库交互。平均每个环境 35 个工具，包括查询类（搜索航班、查找商品）、创建类（创建订单、添加用户）、修改类（更新地址、修改预订）、计算类（计算总价、统计销量）。

第五步：任务生成

LLM 根据环境信息生成需要多轮工具交互才能完成的任务，复杂度从 1-2 轮到 10+ 轮不等，每个任务有明确的完成标准。

第六步：质量验证

验证工具代码可正确执行、数据库状态一致、任务可解，过滤质量不达标的环境。

第七步：强化学习训练

使用 GRPO 算法，从任务池采样，Agent 与环境多轮交互生成轨迹，通过数据库状态验证计算奖励，使用 GRPO 更新策略。

第八步：奖励函数设计

基于数据库状态的奖励验证：任务完成前记录初始状态，完成后查询最终状态，比较是否符合要求。例如任务"预订航班"则检查 flights 表是否新增对应记录。

关键公式/算法解读

▼

GRPO 损失函数

GRPO 的核心思想是在一个批次中采样多条轨迹，通过组内比较来估计优势函数，而不需要单独的价值网络。对于同一任务的多个采样轨迹，根据它们的奖励值进行相对排名——奖励高的轨迹获得正优势，奖励低的获得负优势。这消除了对 baseline/value function 的需求。

数据库状态验证奖励

这是 AWM 最重要的创新之一。奖励不是通过 LLM 判断（不可靠），而是通过实际的数据库查询来验证：

  • 在 Agent 执行任务前，记录数据库的初始状态
  • Agent 完成交互后，查询数据库的最终状态
  • 比较状态变化是否符合任务要求
  • 例如：任务"预订航班"，检查 flights 表中是否新增了对应预订记录

这种方法保证了奖励信号的客观性和可靠性。

实验设计分析

▼

评估基准：论文在三个主流的 tool-use 基准上评估——ToolBench（大规模工具使用基准）、T-Bench（多轮交互式任务基准）、以及第三个分布外基准。这些基准测试了不同能力维度，且与训练环境不完全重叠，能有效评估分布外泛化能力。

对比方法：基础模型（如 Qwen-2.5-7B）、SFT 方法、其他 RL 训练方法、在基准数据上训练的方法。

关键结果：

  • AWM 训练的 Agent 在所有三个基准上都展现了强大的泛化能力
  • 仅在合成环境上训练（不使用基准数据），就超过了在基准数据上训练的方法
  • 环境数量越多泛化能力越强（100 → 500 → 1000）
  • 数据库支撑的环境优于 LLM 模拟的环境
  • 奖励函数设计对性能有显著影响
  • 工具多样性对泛化至关重要

为什么这些结果重要：它们证明了"合成环境训练 → 真实场景泛化"这条路径的可行性，意味着我们可以通过构建更多合成环境来持续提升 Agent 能力。

局限性

▼
  • 虽然生成了 1,000 个环境，但仍然可能无法覆盖所有真实世界的复杂场景，特别是需要领域专业知识的场景
  • 合成生成的数据虽然合理，但可能与真实数据分布存在差异
  • 自动生成的工具相对简单，可能无法模拟真实 API 的复杂行为（如错误处理、并发、延迟等）
  • 环境主要基于英文，对多语言支持的考虑有限
  • 当前的环境主要支持单次任务完成，对长期、持续性的 Agent 交互模拟不足
  • LLM 生成的工具代码可能包含 bug 或边界情况未处理

个人思考

这篇论文在 Agentic RL 领域做出了重要的方法论贡献。"用代码+数据库替代 LLM 模拟环境"的核心洞察解决了训练可扩展性的关键瓶颈。

AWM 管线理论上可以无限扩展环境数量，为持续提升 Agent 能力提供了清晰路径。从"让 LLM 判断"到"查数据库验证"的奖励设计范式转变，使得奖励信号更加可靠。

这种"合成环境 → RL 训练 → 真实场景泛化"的范式可能会成为 Agent 训练的主流方法。
