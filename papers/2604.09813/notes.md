# COVERT: Controllable and Verifiable Tool-Use Data Synthesis for Agentic Reinforcement Learning

## 基本信息

- **标题**: COVERT: Controllable and Verifiable Tool-Use Data Synthesis for Agentic Reinforcement Learning
- **作者**: Siyuan Xu, Shiyang Li, Xin Liu, Tianyi Liu, Yixiao Li, Zhan Shi, Zixuan Zhang, Zilong Wang, Qingyu Yin, Jianshu Chen, Tuo Zhao, Bing Yin (Amazon, Penn State, Georgia Tech)
- **发表日期**: 2026年4月10日
- **arXiv ID**: 2604.09813
- **arXiv 链接**: <https://arxiv.org/abs/2604.09813>
- **领域/关键词**: reinforcement learning, tool use, 轨迹合成, oracle-preserving, 数据增强

## 一句话总结
>
> COVERT 提出了一种两阶段合成数据流水线，先生成可靠的基础工具调用轨迹，再通过"保留最优解"（oracle-preserving）的增强策略系统性地增加环境复杂度，从而为强化学习提供可自动计算奖励的训练环境，显著提升大语言模型在工具调用场景下的鲁棒性。

## 论文精读

### 背景与前置知识

**大语言模型（LLM）工具调用能力**：现代大语言模型不仅需要回答问题，还需要调用外部工具（API、计算器、代码解释器等）来获取实时信息或执行操作。例如，一个模型可能需要调用天气 API 来回答"今天上海下雨吗？"。这被称为函数调用（Function Calling）或工具使用（Tool Use）。

**监督微调（SFT）**：传统上，训练模型使用工具的方法是收集大量"用户提问 -> 工具调用 -> 工具返回结果 -> 最终回答"的示例数据，让模型模仿这些示例。这种方式称为监督微调。

**强化学习（RL）**：与 SFT 不同，RL 让模型自己尝试调用工具，然后根据结果的好坏给予奖励或惩罚。这种方式可以让模型学会更复杂的策略，但它需要一个可以执行工具调用的环境和可靠的奖励信号。

**RLVR（Reinforcement Learning with Verifiable Rewards）**：一种强化学习范式，其中奖励可以通过规则自动计算（例如判断答案是否正确），而不需要人工标注。这种方式特别适合有明确正确/错误答案的场景。

**验证数据合成的挑战**：为 RL 合成工具使用数据面临两难——既要合成足够多样化的场景（包含各种干扰、噪声、错误情况），又要确保知道"正确解"是什么，这样才能自动计算奖励。

### 核心思想详解

COVERT 的核心洞察可以用一个比喻来理解：

**想象你在训练一个厨师（LLM）做菜。** 传统的 SFT 方式是给厨师看菜谱："放盐5克，翻炒3分钟。"厨师照着做就好。但如果要训练厨师应对复杂情况，比如冰箱里有多余的食材（干扰工具）、客人说"我想吃那个红红的圆的东西"（模糊查询）、或者炉子偶尔会熄火（工具输出错误），SFT 就无能为力了——因为菜谱里不会写这些意外情况。

COVERT 的做法是：**先让厨师在简单环境下做菜（基础轨迹生成），确保他会最基本的菜式。然后在一个模拟厨房里，系统性地制造各种意外情况，但厨师手里始终有一份"标准答案"——无论环境怎么变，最正确的做菜步骤和最终菜品质量是已知的。** 这样，当厨师在模拟厨房里练习时，系统可以自动判断他做得好不好：选对了工具吗？在干扰工具中挑出了正确的那个吗？发现炉子坏了知道报告而不是编造结果吗？

核心的"保留最优解"（Oracle-Preserving）原则是：在增加环境复杂度时，确保正确的工具调用步骤（c*）和最终答案（a*）保持不变，这样模型在复杂环境中的表现就可以通过简单的参考匹配来评估。

### 方法逐步拆解

**整体流程分为两个阶段：**

#### 第一阶段：可靠基础轨迹生成

**步骤 1：种子池初始化**。从一个手动精心挑选的小型工具使用轨迹集合开始（种子池），每个轨迹包含：系统提示(s)、用户查询(u)、可用工具集(T)、最优工具调用(c*)、工具输出(o*)、最终答案(a*)。

**步骤 2：自进化生成**。每一轮生成中，从种子池随机采样一个轨迹作为上下文示例，然后让 LLM 根据工具规范、多样性指令和示例来生成新的候选轨迹。通过验证的轨迹会被添加回种子池，形成自进化循环——新验证的轨迹成为后续生成的新示例。

**步骤 3：多层次验证**。每个候选轨迹通过三个检查器的筛选：
- **格式检查器（确定性）**：验证工具定义和调用是否包含必要的结构标签（如 \<tool_call\>），参数是否为有效 JSON。
- **规则检查器（确定性）**：强制工具模式约束——必要参数是否存在，参数类型是否正确，枚举值是否合规。
- **语义检查器（LLM 基础）**：用 LLM 判断工具选择是否符合上下文、参数是否语义合理、最终答案是否与工具输出一致。

**步骤 4：多样性指令**。维护一个异质提示元素池（工具领域、参数类型、参数数量等），每轮随机组合，确保生成的轨迹覆盖广泛的工具使用场景。

#### 第二阶段：Oracle-Preserving 增强

对于每一个通过第一阶段的可靠基础轨迹，COVERT 会应用六种类型的系统化增强：

**增强类型 1：干扰工具**。在工具集中添加多个结构有效、语义相似但与当前任务无关的工具。这迫使模型依赖上下文推理而非表面匹配来选择正确工具。最优调用(c*)和答案(a*)保持不变。

**增强类型 2：查询改写**。将用户问题改写为间接或模糊的表达方式。例如，将"查询 TSLA 股价"改写为"那家造电动汽车的公司今天股票表现如何？"。

**增强类型 3：多格式解析**。将工具输出嵌入到长系统日志中，或改变其呈现格式（如从 JSON 改为 CSV），要求模型准确提取相关信号。

**增强类型 4：噪声输出**。在工具输出中添加冗余信息、错误标记或格式混乱的内容，测试模型的信息提取能力。

**增强类型 5：错误输出（需要 Judge 辅助）**。将工具输出替换为错误消息（如"连接错误"），期望模型检测到异常并适当回应（如报告错误而非编造答案）。这类增强需要 LLM Judge 辅助评估。

**增强类型 6：问题查询**。模拟用户问题与任何工具都无关、或不需要使用工具、或缺少必要参数的情况。期望模型直接回答或请求澄清。

### 关键公式/算法解读

**奖励函数设计**：

```
R(τ) = λ_fmt * r_fmt(τ) + λ_cor * r_cor(τ, m_k)
```

这个公式定义了整体奖励的计算方式：
- `τ`：模型生成的完整轨迹（包含工具调用和最终回答）
- `r_fmt`：格式奖励，检查模型输出是否包含正确的 \<tool_call\> 标签、标签内容是否为有效 JSON
- `r_cor`：正确性奖励，根据增强类型(m_k)的不同采取不同的计算方式
- `λ_fmt` 和 `λ_cor`：两个奖励分量的权重系数

**正确性奖励的两种模式**：

```
r_cor(τ, m_k) = 
  match(τ, c*, a*)   当 m_k = exact 时
  judge(τ, c*, k)    当 m_k = judge-assisted 时
```

- **精确验证（Exact）**：对于干扰工具、查询改写等增强，直接比较模型生成的工具调用与最优解(c*)，以及最终答案与正确答案(a*)。这是确定性的。
- **Judge 辅助验证（Judge-Assisted）**：对于错误输出等增强，用一个轻量级 LLM（实际使用 Qwen2.5-Instruct-7B）来判断模型的行为是否符合预期（例如是否识别到了工具故障）。

**多轮交互的正确性计算**：

```
r_cor(τ, m_k) = (1/T) * Σ r_cor^(t)
```

当涉及多轮工具调用时，对每一轮的正确性取平均。

### 实验设计分析

**为什么选这些数据集？**

论文选择了两个工具使用基准测试：
- **BFCL v3（Berkeley Function Calling Leaderboard）**：最权威的函数调用评测基准，覆盖单轮、多轮、并行调用等场景，以及无关工具和干扰工具的情况。
- **ACEBench**：综合性工具使用评估，包含普通任务、特殊任务（噪声输出、多格式输出）和智能体任务。

同时选择了五个通用能力基准测试来监控 RL 训练是否损害了模型的其他能力：LiveCodeBench（代码）、AIME/Math500/GSM8K（数学推理）、MMLU-Pro（知识）、IFEval（指令遵循）、GPQA（科学推理）。

**模型选择**：选用 Qwen2.5-Instruct-7B 和 14B 作为基础模型，与 TOUCAN-SFT 保持一致以进行公平对比。

**关键结果解读**：

1. **COVERT-RL 对工具能力的提升**：在 14B 模型上，BFCL v3 从 56.5 提升到 59.9（+3.4），ACEBench 从 53.0 提升到 59.3（+6.3）。最大提升出现在论文增强目标对应的类别上——BFCL 的无关检测（+11.9）、ACEBench 的特殊任务（+14.2，涵盖噪声/多格式/错误输出）。

2. **SFT 与 RL 的互补性**：先做 SFT 再做 RL（即 TOUCAN-SFT + COVERT-RL）取得了最佳结果：BFCL 62.1，ACEBench 61.8。这说明 SFT 提供结构先验，RL 强化鲁棒性，两者互补。

3. **通用能力保持**：COVERT-RL 在非工具基准测试上几乎没有退化（最大下降仅 1.0 点），而 TOUCAN-SFT 在某些任务上出现了显著退化（如 AIME24 从 20.0 降到 10.0）。

4. **局限之处**：多轮交互和智能体任务未获得提升，这是符合预期的——当前模拟环境主要针对短到中等长度的交互，尚未模拟紧密耦合的状态依赖。

## 问题与动机

现有合成工具使用数据主要面向监督微调（SFT），每个样本是静态的"问题-工具调用-回答"三元组。然而，强化学习（RL）需要可执行的环境和可自动计算的奖励信号。现有方法要么使用有限的真实执行器（如搜索引擎）、要么在已有的 SFT 数据上跑 RL，都无法系统性地控制环境复杂度（如干扰工具、模糊查询、噪声输出）。COVERT 的目标是构建一个 RL 就绪的合成数据流水线，既能大规模生成多样化的工具使用场景，又能自动计算奖励信号。

## 核心方法

COVERT 是一个两阶段流水线：**阶段一**通过自进化生成和多层次验证产生可靠的基础工具使用轨迹（简单工具、明确查询、直接输出）；**阶段二**应用"oracle-preserving"增强策略，系统性地修改环境组件（系统提示、用户查询、工具输出），引入干扰工具、间接查询、多格式/噪声/错误工具输出等，同时保留最优工具调用和最终答案作为奖励计算的参考。对于标准情况通过参考匹配自动计算奖励，对于特殊行为（如错误检测）则使用轻量级 LLM Judge 辅助验证。RL 训练采用 GRPO 算法。

## 实验结果

- **数据集**: BFCL v3, ACEBench（工具使用基准）；LiveCodeBench, AIME, Math500, GSM8K, IFEval, GPQA, MMLU-Pro（通用能力基准）
- **主要指标**: 工具调用准确率（Overall Accuracy），通用基准准确率
- **关键结果**: Qwen2.5-Instruct-14B 上，COVERT-RL 提升 BFCL v3 从 56.5 到 59.9，ACEBench 从 53.0 到 59.3；SFT+RL 堆叠进一步达到 62.1 和 61.8；通用能力几乎无退化
- **对比方法**: Qwen2.5-Instruct（基础模型）, TOUCAN-SFT（SFT 基线）, TOUCAN-SFT + COVERT-RL（堆叠）
  - 与 ToolRL、ToolZero 不直接比较（训练数据、工具集、提示协议不同）

## 局限性

1. **多轮交互能力有限**：当前的模拟环境主要针对短到中等长度的交互，未模拟紧密耦合的状态依赖关系，导致多轮和智能体任务上没有提升。
2. **Judge 辅助验证非确定性**：部分增强类型依赖 LLM Judge 进行验证，Judge 的决策不是确定性的，目前尚未进行系统的人类一致性研究。
3. **环境复杂度可控但有限**：当前的扰动类型覆盖了六种，但真实世界的工具使用场景可能包含更复杂的交互模式。
4. **与其他方法隔离比较**：由于训练数据、工具集和提示协议的差异，未与 ToolRL 和 ToolZero 进行直接比较。
5. **通用能力的轻微退化**：虽然在可接受范围内，但在某些基准上（如 14B 的 GPQA-Diamond -1.0 点）存在轻微下降。

## 个人思考

COVERT 的核心贡献在于提出了"oracle-preserving"这个设计原则——将环境扰动与标准答案解耦，使得在复杂环境中也能自动计算奖励。这一思路在各种 RL 训练场景中都有借鉴意义，不仅限于工具调用。

值得注意的是，COVERT-RL 在提升工具鲁棒性的同时几乎没有损害通用能力，这与 SFT 在大规模工具数据上微调时常出现的通用能力退化形成鲜明对比。这表明 RL 作为一种后期精调策略，比 SFT 更适合在特定技能和通用能力之间取得平衡。

关于多轮任务没有提升这一点，我认为根本原因是"oracle-preserving"在多轮场景中面临挑战——轮次越多，每步的"最优动作"越难定义，因为前面步骤的选择会影响后面的状态。如果未来能扩展到多轮场景，可能需要定义分层次的 oracle（如子目标级别的参考信号）。

此外，COVERT 使用 Qwen2.5-Instruct-7B 作为 Judge 模型来评估 14B 模型的输出，这种"小模型判断大模型"的做法在计算成本上很经济，但其可靠性可能需要更多验证。
