COVERT: Controllable and Verifiable Tool-Use Data Synthesis for Agentic Reinforcement Learning

Siyuan Xu, Shiyang Li, Xin Liu, Tianyi Liu, Yixiao Li, Zhan Shi, Zixuan Zhang, Zilong Wang, Qingyu Yin, Jianshu Chen, Tuo Zhao, Bing Yin 2026-04-10
arXiv:2604.09813
reinforcement learning tool use data synthesis oracle-preserving LLM

一句话总结

COVERT 提出了一种两阶段合成数据流水线，先生成可靠的基础工具调用轨迹，再通过"保留最优解"（oracle-preserving）的增强策略系统性地增加环境复杂度，从而为强化学习提供可自动计算奖励的训练环境，显著提升大语言模型在工具调用场景下的鲁棒性。

背景与前置知识

▼

大语言模型（LLM）工具调用能力：现代大语言模型不仅需要回答问题，还需要调用外部工具（API、计算器、代码解释器等）来获取实时信息或执行操作。例如，一个模型可能需要调用天气 API 来回答"今天上海下雨吗？"。

监督微调（SFT）：传统上，训练模型使用工具的方法是收集大量"用户提问 -> 工具调用 -> 工具返回结果 -> 最终回答"的示例数据，让模型模仿这些示例。

强化学习（RL）：与 SFT 不同，RL 让模型自己尝试调用工具，然后根据结果的好坏给予奖励或惩罚。这种方式可以让模型学会更复杂的策略，但它需要一个可以执行工具调用的环境和可靠的奖励信号。

RLVR（Verifiable Rewards RL）：一种强化学习范式，其中奖励可以通过规则自动计算（例如判断答案是否正确），而不需要人工标注。

验证数据合成的挑战：为 RL 合成工具使用数据面临两难——既要合成足够多样化的场景（包含各种干扰、噪声、错误情况），又要确保知道"正确解"是什么，这样才能自动计算奖励。

核心思想详解

▼

COVERT 的核心洞察可以用一个比喻来理解：想象你在训练一个厨师（LLM）做菜。传统的 SFT 方式是给厨师看菜谱，照着做就好。但如果要训练厨师应对复杂情况——冰箱里有多余的食材（干扰工具）、客人说"我想吃那个红红的圆的东西"（模糊查询）、炉子偶尔会熄火（工具输出错误）——SFT 就无能为力了。

COVERT 的做法是：先让厨师在简单环境下做菜（基础轨迹生成），确保他会最基本的菜式。然后在一个模拟厨房里，系统性地制造各种意外情况，但厨师手里始终有一份"标准答案"——无论环境怎么变，最正确的做菜步骤和最终菜品质量是已知的。 这样，当厨师在模拟厨房里练习时，系统可以自动判断他做得好不好。

核心的"保留最优解"（Oracle-Preserving）原则是：在增加环境复杂度时，确保正确的工具调用步骤（c*）和最终答案（a*）保持不变，这样模型在复杂环境中的表现就可以通过简单的参考匹配来评估。

方法逐步拆解

▼

第一阶段：可靠基础轨迹生成

步骤 1：种子池初始化。 从一个手动精心挑选的小型工具使用轨迹集合开始（种子池），每个轨迹包含：系统提示(s)、用户查询(u)、可用工具集(T)、最优工具调用(c*)、工具输出(o*)、最终答案(a*)。

步骤 2：自进化生成。 每轮从种子池随机采样一个轨迹作为上下文示例，让 LLM 根据工具规范、多样性指令和示例生成新的候选轨迹。通过验证的轨迹被添加回种子池。

步骤 3：多层次验证。 每个候选轨迹通过三个检查器：格式检查器（检查结构标签和 JSON 有效性）、规则检查器（检查参数约束和类型）、语义检查器（LLM 判断工具选择是否合理）。

步骤 4：多样性指令。 维护一个异质提示元素池（工具领域、参数类型、参数数量等），每轮随机组合，确保覆盖广泛的工具使用场景。

第二阶段：Oracle-Preserving 增强

六种增强类型：（1）干扰工具——添加语义相似但无关的工具；（2）查询改写——变为间接模糊表达；（3）多格式解析——工具输出嵌入日志或改变格式；（4）噪声输出——添加冗余/错误标记；（5）错误输出——工具返回错误消息，期望模型检测异常（需 Judge 辅助）；（6）问题查询——模拟无关、不需要工具或缺少参数的查询。

关键公式/算法解读

▼

奖励函数：R(τ) = λ_fmt · r_fmt(τ) + λ_cor · r_cor(τ, m_k)

整体奖励 = 格式奖励（检查标签和JSON有效性）+ 正确性奖励（根据增强类型计算）。

两种正确性验证模式：

  • 精确验证（Exact）：直接比较模型输出与最优解(c*)和正确答案(a*)，确定性的参考匹配。
  • Judge 辅助验证（Judge-Assisted）：用轻量级 LLM（Qwen2.5-Instruct-7B）判断模型行为是否符合特定能力目标。

多轮交互：r_cor = (1/T) * Σ r_cor^(t)，对每一轮的正确性取平均。

实验设计分析

▼

基准选择：BFCL v3（最权威的函数调用评测）和 ACEBench（综合性工具使用评估），覆盖单轮、多轮、并行调用及干扰工具场景。

通用能力监控：LiveCodeBench（代码）、AIME/Math500/GSM8K（数学）、MMLU-Pro（知识）、IFEval（指令遵循）、GPQA（科学推理）。

模型与基线：Qwen2.5-Instruct-7B/14B 为基础模型，对比 TOUCAN-SFT 基线。

关键结果：

  • 14B 上 BFCL v3 +3.4，ACEBench +6.3
  • 最大提升在增强目标类别：无关检测 +11.9，特殊任务 +14.2
  • SFT+RL 堆叠取得最佳效果，证明两者互补
  • 通用能力几乎无退化（最大 -1.0），而 SFT 基线退化严重
  • 多轮和智能体任务未提升，符合预期

局限性

▼
  • 多轮交互能力有限：模拟环境主要针对短到中等长度交互，未模拟紧密耦合的状态依赖关系。
  • Judge 辅助验证非确定性：部分增强依赖 LLM Judge，其决策非确定，尚未进行系统的人类一致性研究。
  • 环境复杂度覆盖有限：当前六种扰动能覆盖常见场景，但真实世界可能包含更复杂的交互模式。
  • 与其他方法隔离比较：因训练数据、工具集和提示协议的差异，未与 ToolRL/ToolZero 直接比较。
  • 通用能力轻微退化：虽然可接受，但在某些基准存在轻微下降。

个人思考

COVERT 的核心贡献在于"oracle-preserving"设计原则——将环境扰动与标准答案解耦，使得在复杂环境中也能自动计算奖励。该思路对各种 RL 训练场景都有借鉴意义。值得注意的是，COVERT-RL 在提升工具鲁棒性的同时几乎没有损害通用能力，与 SFT 的退化形成鲜明对比，表明 RL 作为后期精调策略更适合在特定技能和通用能力间取得平衡。多轮任务未提升的根因在于——轮次越多，"最优动作"越难定义。未来扩展到多轮场景可能需要分层 oracle。
