# Large-Scale Terminal Agentic Trajectory Generation from Dockerized Environments

## 基本信息

- **标题**: Large-Scale Terminal Agentic Trajectory Generation from Dockerized Environments
- **作者**: Siwei Wu, Yizhi Li, Yuyang Song, Wei Zhang, Yang Wang, Riza Batista-Navarro, Xian Yang, Mingjie Tang, Bryan Dai, Jian Yang, Chenghua Lin
- **发表日期**: 2026-02-04
- **arXiv ID**: 2602.01244
- **arXiv 链接**: <https://arxiv.org/abs/2602.01244>
- **领域/关键词**: Terminal Agent, Trajectory Generation, Dockerized Environment, Reinforcement Learning, Code Generation, Test-time Scaling

## 一句话总结
>
> TerminalTraj 提出了一套从 Docker 化环境中大规模生成终端智能体轨迹的完整流水线，通过预测性质量过滤和可执行验证代码，从近百万个 GitHub 仓库中筛选并生成了超过 5 万条高质量的多轮交互轨迹，训练出的 TerminalTraj-32B 模型在 TerminalBench 1.0 和 2.0 上分别取得了超过骨干模型 20 分和 10 分的提升。

## 论文精读

### 背景与前置知识

**终端智能体（Terminal Agent）**：一类能够通过命令行界面（shell）与计算机环境交互的 AI 智能体。与传统的代码生成模型不同，终端智能体需要理解环境状态、执行命令、读取输出并根据反馈动态调整策略。典型代表有 OpenHands、SWE-Agent 等。

**TerminalBench（TB）**：专门用于评估终端智能体性能的基准测试。TB 1.0 包含 46 个 Docker 化任务，TB 2.0 扩展到 72 个任务，涵盖多模态、SQL、Web 服务、安全等多个领域。评估采用 pass@k 指标，即 k 次尝试中至少一次通过测试的概率。

**轨迹（Trajectory）**：智能体在执行任务过程中产生的完整交互序列，包括环境状态、智能体的动作（命令）、环境反馈等多轮对话记录。高质量的轨迹是训练智能体的关键训练数据。

**Docker 化环境**：将任务运行环境打包为 Docker 镜像，确保环境的可复现性和隔离性。每个任务对应一个 Docker 镜像，包含所需的依赖、数据文件和初始状态。

**多轮监督微调（Multi-turn SFT）**：与单轮问答不同，终端智能体任务需要多轮交互。训练数据格式为多轮对话序列，模型需要在每一轮根据历史上下文和当前环境状态生成下一步动作。

### 核心思想详解

TerminalTraj 的核心思想可以用一个"淘金"的比喻来理解：互联网上有海量的开源项目和代码仓库，但绝大多数并不适合用来训练终端智能体。关键挑战在于如何从"沙子"中筛出"金子"——即那些环境可复现、任务有意义、轨迹可验证的高质量训练数据。

整个流水线的设计遵循一个层层递进的漏斗逻辑：

1. **广撒网**：从 GitHub 等平台收集近 90 万个代码仓库作为原始数据源
2. **粗筛**：构建 Docker 镜像，用 ScoreModel 预测哪些镜像更有可能产出高质量数据
3. **精炼**：为通过筛选的镜像生成任务实例（包含任务描述和验证代码），用大模型（Qwen3-Coder-480B）在环境中执行任务，产生交互轨迹
4. **验真**：通过可执行的 pytest 验证代码检验轨迹的正确性，只保留通过验证的轨迹

这套流水线的巧妙之处在于它解决了一个核心矛盾：高质量轨迹的生成需要大量计算资源（每条轨迹都需要在 Docker 容器中实际运行），但原始数据中只有极小比例能产出可用数据。通过引入 ScoreModel 做预测性过滤，将 Docker 构建成功率从随机选择的 9% 提升到了 19.5%，大幅提高了资源利用效率。

### 方法逐步拆解

**Step 1: 数据源收集（Data Sources Collection）**

- 从 GitHub 收集 899,741 个代码仓库，覆盖 8 个领域（多模态、SQL、QEMU、Web 服务、安全、数据处理、模型训练与评估、环境交互）和 8 种编程语言
- 领域分类通过仓库的 README、文件结构等元信息自动判定
- 最终收集到 1,265,972 个数据源实例

**Step 2: Docker 镜像策展（Docker Image Curation）**

- 为每个代码仓库生成 Dockerfile 并构建镜像
- 引入 ScoreModel（基于奖励模型训练的质量预测器）对镜像进行质量评分
- 质量评分公式：Q_i = Σ ScoreModel(f_j) / N_i，其中 f_j 是镜像中的文件，N_i 是文件数量
- 设定阈值 τ=0.2，只有 Q_i > τ 的镜像才进入下一阶段
- 通过 ScoreModel 筛选后，Docker 构建成功率达到 19.5%，而随机选择仅 9%
- 最终成功构建 32,325 个 Docker 镜像

**Step 3: 实例生成（Instance Generation）**

- 对每个通过筛选的 Docker 镜像，生成任务描述（task query）和可执行验证代码（executable validation code）
- 任务描述使用大模型自动生成，描述需要完成的目标
- 验证代码是 pytest 测试套件，用于自动检验任务是否正确完成
- 每个镜像可生成多个不同难度的任务实例
- 最终生成 1,030,695 个任务实例

**Step 4: 轨迹滚动与验证（Trajectory Rolling and Verification）**

- 使用 Terminus-2 脚手架（scaffold）作为智能体与环境的交互框架
- 采用 Qwen3-Coder-480B 作为轨迹生成的"滚动模型"（rolling model）
- 每个任务实例在对应 Docker 容器中运行，记录完整的交互轨迹
- 轨迹验证：运行 pytest 验证代码，只有全部测试通过的轨迹才被保留
- 最终收集到 50,733 条经过验证的高质量轨迹

**Step 5: 模型训练**

- 使用 Qwen2.5-Coder 系列作为骨干模型（7B、14B、32B 三个规模）
- 采用 Megatron-LM 框架进行多轮监督微调（multi-turn SFT）
- 训练数据格式为多轮对话序列，模型学习在每个交互轮次中根据环境状态生成正确的命令

### 关键公式/算法解读

**质量评分公式**：

Q_i = Σ ScoreModel(f_j) / N_i

- Q_i：第 i 个 Docker 镜像的质量分数
- f_j：镜像中的第 j 个文件
- N_i：第 i 个镜像中的文件总数
- ScoreModel：一个训练好的奖励模型，输入文件内容，输出该文件对构建质量的贡献分数
- 阈值 τ=0.2：只有质量分数超过此阈值的镜像才被保留

这个公式的直觉是：一个高质量的 Docker 镜像应该包含"有价值"的文件（如配置良好的依赖、有意义的代码逻辑），而不是随机拼凑的文件。ScoreModel 学习了什么样的文件特征与高质量环境相关。

**Pass@k 指标**：

pass@k = 1 - C(n-c, k) / C(n, k)

- n：总尝试次数
- c：成功次数
- k：允许的尝试次数
- 该指标衡量在 k 次独立尝试中至少成功一次的概率，是评估代码生成和智能体性能的标准指标

**Test-time Scaling**：

通过增加推理时的采样次数（k 值），观察模型性能随 k 的增长趋势。TerminalTraj 训练的模型在 pass@k 曲线上表现出更好的缩放特性，说明训练数据质量对推理阶段的性能提升有显著帮助。

### 实验设计分析

**基准测试选择**：选用 TerminalBench 1.0（46 个任务）和 TerminalBench 2.0（72 个任务）作为评估基准，这两个基准专门针对终端智能体设计，涵盖多个领域和不同难度级别，具有较强的代表性和挑战性。

**对比方法**：与多种方法对比，包括骨干模型（Qwen2.5-Coder 系列）、基于现有数据集训练的模型（Nex-N1、SETA）以及其他开源智能体框架。

**消融实验**：通过逐步去除流水线中的关键组件（ScoreModel 筛选、验证代码等），验证每个组件的贡献。

**主要实验发现**：
- TerminalTraj-32B 在 TB 1.0 上达到 77.08%（pass@1），比骨干模型提升 20+ 分
- TerminalTraj-32B 在 TB 2.0 上达到 63.89%（pass@1），比骨干模型提升 10+ 分
- 在 pass@8 设置下，TerminalTraj-32B 在 TB 1.0 上达到 87.50%，在 TB 2.0 上达到 76.39%
- 与 Nex-N1 数据集训练的模型相比，TerminalTraj 在 TB 1.0 上有 1.7 倍的提升，在 TB 2.0 上有 1.3 倍的提升
- 与 SETA 相比，pass@8 从 12.5% 提升到 45%

## 问题与动机

当前终端智能体的发展面临一个核心瓶颈：**高质量训练数据的匮乏**。

传统的代码生成数据集（如 HumanEval、MBPP）只包含单轮的"问题-代码"对，而终端智能体需要的是多轮交互式的轨迹数据——包含环境状态、命令执行、错误处理和动态策略调整的完整序列。这种数据的获取成本极高，因为每条轨迹都需要在真实环境中执行和验证。

此外，现有的一些轨迹数据集存在以下问题：
- **环境不可复现**：依赖特定的系统配置或外部服务，无法在隔离环境中重现
- **缺乏可执行验证**：没有自动化的方式验证轨迹是否真正完成了任务
- **数据规模有限**：人工标注或小规模生成的轨迹数据难以满足大模型训练的需求
- **领域覆盖不均**：多集中在软件开发领域，缺少安全、系统管理、数据处理等多样化场景

TerminalTraj 通过 Docker 化环境解决了可复现性问题，通过 pytest 验证代码解决了自动化验证问题，通过大规模自动化流水线解决了数据规模问题，通过多领域数据源解决了覆盖不均问题。

## 核心方法

TerminalTraj 的核心是一个四阶段的自动化数据生成流水线：

1. **数据源收集与分类**：从 GitHub 大规模爬取代码仓库，按 8 个领域和 8 种编程语言自动分类

2. **预测性 Docker 镜像策展**：不是盲目地为所有仓库构建 Docker 镜像（成功率极低），而是先用 ScoreModel 预测哪些仓库更可能成功构建。ScoreModel 从文件内容和结构层面评估质量，将构建成功率从 9% 提升到 19.5%，节省了大量计算资源

3. **任务实例生成**：对每个通过筛选的镜像，自动生成任务描述和对应的 pytest 验证代码。验证代码是流水线的关键创新之一——它使得轨迹质量的评估完全自动化，无需人工介入

4. **大规模轨迹滚动**：使用 Qwen3-Coder-480B 在 Terminus-2 框架下执行任务，记录完整的多轮交互轨迹，并通过 pytest 自动验证。最终从超过 100 万个任务实例中筛选出约 5 万条通过验证的轨迹

这套方法的设计哲学是"以环境为中心"：不是先定义任务再找环境，而是先从真实代码仓库中提取环境，再基于环境生成任务。这使得数据更加贴近真实世界的使用场景。

## 实验结果

- **数据集**: TerminalBench 1.0（46 个任务）、TerminalBench 2.0（72 个任务）
- **主要指标**: pass@1、pass@2、pass@4、pass@8
- **关键结果**:
  - TerminalTraj-32B 在 TB 1.0 pass@1 达到 77.08%，TB 2.0 达到 63.89%
  - 相比骨干模型 Qwen2.5-Coder-32B，TB 1.0 提升 20+ 分，TB 2.0 提升 10+ 分
  - pass@8 下 TB 1.0 达 87.50%，TB 2.0 达 76.39%
  - 与 Nex-N1 数据集相比，TB 1.0 提升 1.7 倍，TB 2.0 提升 1.3 倍
  - 与 SETA 相比，TB 2.0 pass@8 从 12.5% 提升到 45%
  - ScoreModel 筛选使 Docker 构建成功率从 9% 提升至 19.5%
  - 流水线从 899,741 个仓库中最终产出 50,733 条验证轨迹
- **对比方法**: Qwen2.5-Coder（7B/14B/32B）、Nex-N1 训练模型、SETA、OpenHands、SWE-Agent 等

## 局限性

1. **计算成本高昂**：整个流水线需要构建数万个 Docker 镜像并在其中运行大规模轨迹生成，计算资源消耗巨大。论文未详细报告总计算成本，但可以推测需要大量的 GPU 和 CPU 资源。

2. **Docker 构建成功率仍然较低**：即使使用 ScoreModel 筛选，Docker 构建成功率仍仅 19.5%，意味着约 80% 的计算资源被浪费。如何进一步提高构建成功率是一个开放问题。

3. **轨迹质量依赖滚动模型**：轨迹的生成质量在很大程度上取决于 Qwen3-Coder-480B 的能力。如果滚动模型在某些领域表现不佳，生成的轨迹可能存在系统性偏差或遗漏。

4. **验证代码的覆盖度**：pytest 验证代码只能检查预定义的测试用例，无法保证任务的完整正确性。可能存在"通过了测试但实际解决方案不正确"的情况。

5. **领域覆盖的均衡性**：虽然覆盖了 8 个领域，但从附录数据来看，各领域的数据量分布不均匀，某些领域（如多模态、SQL）的数据量可能远大于其他领域。

6. **语言覆盖有限**：仅覆盖 8 种编程语言，对于一些重要的语言（如 Rust、Go、TypeScript 等）可能覆盖不足。

## 个人思考

TerminalTraj 的工作非常有启发性，它代表了数据工程在 AI 训练中日益重要的趋势。几个值得深思的方向：

**数据工程 > 模型架构**：TerminalTraj 的模型架构并未做特殊创新（基于 Qwen2.5-Coder 进行 SFT），但通过精心设计的数据生成流水线，实现了显著的性能提升。这印证了"数据是新的代码"这一观点——在大模型时代，如何高效地生产高质量训练数据可能比模型架构的微调更为关键。

**自动化验证是核心**：流水线中最关键的设计是可执行的验证代码。它将轨迹质量评估从人工审核转变为自动化验证，使得大规模数据生产成为可能。这一思路可以推广到其他领域：只要有可靠的自动化验证手段，就可以大规模生产训练数据。

**ScoreModel 的启示**：用一个小模型指导大模型的数据生成，是一种高效的资源分配策略。这种"模型指导数据生产"的思路可能成为未来数据工程的标准范式。

**与强化学习的结合**：当前 TerminalTraj 只使用了 SFT 训练，没有结合强化学习。如果在 SFT 的基础上进一步使用 RL（如 GRPO 或 PPO），可能会获得更大的性能提升。论文在结论中也提到了这一点作为未来方向。

**从终端到通用交互**：TerminalTraj 的流水线设计虽然针对终端环境，但其核心思想（环境可复现 + 自动化验证 + 预测性筛选）可以迁移到其他类型的智能体数据生成中，例如 Web 交互、API 调用、数据库操作等。
